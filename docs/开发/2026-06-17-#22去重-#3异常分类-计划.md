# 计划:#22 in-flight 去重(只堵错开传)+ #3 异常分类纪律(只改上层)

> 2026-06-17。承接 P4 小硬化后的 backlog。经 /plan-eng-review(含 codex 外部声音)审定。
> 用户拍板:#22=修法一(开工前查缓存,不建去重 index);#3=只改上层接收端分类(底层引擎不动);
> DRY=把 3 处缓存命中出口的重复逻辑合并(碰 websocket_handler)。

## 背景与决策依据

- **真实并发模型**:FunASR Mac 路径并发 = `max_concurrent_tasks`(常用 3),起 N 个 worker;
  worker 取出任务后 `_process_task` **直接转录,中间不查缓存**;`submit_task` 查一次缓存且在
  `_queue_lock` 之外。**另有两个 websocket 层缓存命中早返回出口**(整文件 `:228-266`、分片 `:727-760`),
  在 `create_task` 后、`submit_task` 前命中即直接 `task_complete` 返回,根本不入队。
- **#22 要堵的浪费**:同一文件先后上传两次(同 file_hash)。第二份进来时第一份还没转完,
  `submit_task` 查缓存扑空 → 入队;轮到它时第一份早已写缓存,但 `_process_task` 不查缓存 →
  **从头重转一遍**,白占一个并发槽。
- **决策**:
  - #22 = **修法一**(开工前再查一次缓存)。只堵"错开到达";"几乎同时到达 + 多 worker 并发取出"
    堵不住(已知局限,pool=3 有富余槽可接受)。**关键**:prior learning `inflight-dedup-coupling`
    证明完整去重要建 index 且在 6+ 终态点维护,漏一处→脏 index→轮询命中死 task_id;修法一**不建 index**,
    天然避坑。完整并发去重 = TODOS #22(独立大 PR,本轮不做)。
  - #3 = **只改上层接收端**(task_manager 建 `ErrorKind` 枚举 + `classify_error()` 单一分类器,
    isinstance 优先、字符串兜底)。底层 `funasr_transcriber` / `qwen3_transcriber` 抛类型化异常
    = TODOS #3 完整版的发送端,**本轮不做**(碰两引擎核心,blast radius 大)。
  - DRY = 合并 3 处缓存命中出口的**重复纯逻辑**(metadata 组装 + projected 提取 + SRT-dict 校验)。

## 一、DRY 合并:3 处缓存命中出口的重复纯逻辑(碰 websocket_handler)

### 新增共享纯函数(`src/core/result_projection.py`,已有 `build_result_metadata` 同模块)

```python
def cache_hit_metadata(cached_result, *, engine, options, output_format):
    """3 处缓存命中出口共享:projected 提取 + metadata 构建 + SRT-dict 有效性判断。
    无副作用(不改 cached_result,统一用 get 不 pop)。返回 (metadata, projected, srt_ok)。
    - JSON: srt_ok=True;projected 读 cached_result.metadata
    - SRT : srt_ok = isinstance(dict) and format==srt;projected 读 cached_result["projected"]
    """
```

3 处出口改调它,**控制流各自保留**(codex #7/#8):
- `websocket_handler:228-266`(整文件)& `:727-760`(分片):调 `cache_hit_metadata` 拿 metadata;
  组 result_data 时**排除 projected key**(替代原 `pop`,避免泄漏给客户端);srt_ok=False → 跳过缓存。
- `task_manager.submit_task`:调它拿 metadata,挂 `task.result.metadata`(SRT 仍建 segments=[] 的
  TranscriptionResult + set `task.srt_content`,保持现状语义)。

### 风险与测试
- **回归钉**:3 处缓存命中(SRT diarize=true/false + JSON)行为字节级不变,尤其 SRT projected
  flag 进 metadata、不泄漏给客户端、segments=[] 语义保持(prior learning `qwen3-srt-cache-empty-bug`)。
- parity 必跑(碰 task_manager + websocket_handler)。

## 二、#22 修法一:worker 开工前二次查缓存

### 改动点(task_manager.py)

1. **抽 helper `_populate_task_from_cache(task, cached_result)`**:把 `submit_task` 命中分支
   现有"组装 task.result/srt_content + metadata(调 `cache_hit_metadata`)"逻辑抽出,**字节级等价**。

2. **抽 helper `async _try_complete_from_cache(task) -> bool`**:
   - `if task.force_refresh: return False`。
   - `cache_params_for(task)` → `db_manager.get_cached_result(...)`;未命中 → `return False`。
   - 命中 → `_populate_task_from_cache` → **`task.progress=100`(codex #2)** →
     **`task.error=None`(codex #3)** → `task.status=COMPLETED` → `completed_at` →
     `_record_terminal(COMPLETED)` → `_notify_task_complete` → log → `return True`。

3. **`submit_task` 改造**:命中块替换为 `if await self._try_complete_from_cache(task): return None`。

4. **`_process_task` 开工前二次查缓存**(核心新增,**放进 try 内防泄漏 codex #1**):
   ```python
   async with self._queue_lock:
       self.processing_tasks += 1
   try:
       # #22(只堵错开传): 同 hash 前序任务在本任务排队期间已转完写缓存 → 命中即秒返回。
       # 必须在 try 内: 查缓存自身抛错绝不逃逸(否则任务卡 PENDING 永久泄漏, codex #1)。
       try:
           if await self._try_complete_from_cache(task):
               await self._maybe_delete_task_file(task)
               return
       except Exception as e:
           logger.warning(f"开工前查缓存失败, 继续转录 {task_id}: {e}")  # 降级: 照常真跑
       task.status = TaskStatus.PROCESSING
       ...
   finally:
       async with self._queue_lock:
           self.processing_tasks = max(0, self.processing_tasks - 1)
   ```
   注:命中走缓存路径时该任务已消费一个 worker dequeue slot,处理窗口内 metrics 既不显示
   processing 也不显示 queue(codex #4,通常极短可接受)。

### 测试(TDD 先红)
- 错开到达:get_cached_result 第一次(submit)→ None,第二次(process)→ cached;断言
  `transcriber.transcribe` **未调** + COMPLETED + progress=100 + error=None + `_record_terminal` 记一次。
- **codex #1 防泄漏**:二次查缓存抛异常 → 任务**仍走转录**(不卡 PENDING)。
- force_refresh=True:两处都不查(测试**仅实例化 TaskManager**,codex #10)。
- **word-align 降级 miss(codex #9)**:第一个任务 +wa 失败降级写裸 tag,第二个 +wa 请求二次查 miss → 仍转录。
- helper 等价回归:见第一节(SRT/JSON 3 形态)。

## 三、#3 只改上层:task_manager 错误分类

### 改动点(task_manager.py,ErrorKind 与 QueueFullError 同文件,不碰 websocket import)

1. **`ErrorKind(str, Enum)`** + **每 kind 显式 retryable(codex #11)**:
   ```python
   class ErrorKind(str, Enum):
       QUEUE_FULL = "queue_full"
       TIMEOUT = "timeout"
       NON_RETRYABLE_INPUT = "non_retryable_input"  # 音频太短/格式/文件不存在/太大/认证
       MODEL_ERROR = "model_error"                   # VAD/index/dimension → 可重试 + 重置模型
       ENGINE_ERROR = "engine_error"                 # 其它转录异常, 可重试
   _RETRYABLE = {ErrorKind.MODEL_ERROR, ErrorKind.ENGINE_ERROR}  # 显式集合, 非取反
   # retryable / is_model 用显式集合判定, 不写 "not NON_RETRYABLE_INPUT"
   ```

2. **`classify_error(exc) -> ErrorKind`**:isinstance(QueueFullError)→QUEUE_FULL(保留;注:worker
   catch 里 dead,submit 用其 kind,codex #12);字符串兜底(底层未类型化,过渡)→ NON_RETRYABLE_INPUT
   / MODEL_ERROR / ENGINE_ERROR。`_NON_RETRYABLE_MARKERS` / `_MODEL_ERROR_RE` 模块级常量沿用现有串。

3. **worker catch 改用**(`_process_task`):`kind = classify_error(e)` → `record_error(kind.value)` →
   `should_retry = kind in _RETRYABLE` → `if kind.is_model and task.engine == "funasr": await self._try_reset_model()`
   (**多引擎安全 codex #13**:仅 funasr 任务才 reset funasr)。

4. **删 `_should_retry_error` / `_is_model_error`**(逻辑并入 `classify_error`;删前 grep 确认无其它引用)。

5. **`record_error` 调用点统一枚举值**:submit_task `"queue_full"` → `ErrorKind.QUEUE_FULL.value`;
   看门狗 `"timeout"` → `ErrorKind.TIMEOUT.value`。`record_error` 仍收 str(传 .value),兼容 websocket
   层暂留字符串调用(范围二)。

### 影响 & 测试
- `errors_total{kind}` 新增 `non_retryable_input` / `model_error`(原全归 engine_error)。CLAUDE.md kind 约定同步。
- 测试:classify_error 各分支(QueueFullError→QUEUE_FULL;中文串→NON_RETRYABLE_INPUT retryable=False;
  VAD/index→MODEL_ERROR is_model=True;其它→ENGINE_ERROR);worker 失败路径各 kind retry 行为;
  **qwen3 任务 model_error 不调 `_try_reset_model`(codex #13)**;`errors_total` kind 累加正确。
