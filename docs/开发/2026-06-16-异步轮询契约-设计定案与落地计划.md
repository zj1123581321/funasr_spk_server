> 评审：`/plan-ceo-review`（2026-06-16，SELECTIVE EXPANSION）+ 对抗 spec 子代理（10 findings）+ codex 外部声音（14 findings，含 1 个战略级简化）。三轮已并入。承接止血 PR（5f7eabf→9b7eec7）。
> 定位：根治 `接收消息超时(300s)` 的客户端侧根因。**服务端上传路径零协议改动；异步=客户端收 ack 后轮询而非堵等。**

## 0. TL;DR 定案

| 维度 | 定案 | 理由 |
|---|---|---|
| 实现方案 | **A 薄轮询**（非持久任务表 B / 非双模 C） | 服务端资产几乎现成，最小 diff |
| 兼容策略 | **无 `wait` 开关、无 `task_accepted`**（codex 战略简化） | 服务端入队后**今天就**立即回 `task_queued`/`upload_complete`（带 position）；客户端在此 ack 后改轮询即可。上传协议零改动 ⇒ 无 flag day 伪命题 |
| 批量查询 | **新增 `task_status_batch`，内联 result** | 防 N+1 拉取风暴 + TTL race（codex） |
| 在途去重 | **拆独立 PR**（不在本轮，/plan-eng-review 定案） | 真实耦合远超「加个 dict」：6+ 处终态点漏删风险 + 跨层连接迁移 + canonical-id 契约；只防「白跑一个槽」无正确性 bug，不值与超时根治绑发 |
| 确定性 task_id | **跳过** | 随机 uuid + 重投已能恢复 |
| 限流 | **推迟 TODOS** | 仅 2 可信自控客户端 |
| 吞吐 | **不在 #20 范围** | pool=1 单 GPU 物理上限，正交 |

**核心洞察（codex 战略发现）**：服务端从来不阻塞——入队即回 `task_queued{position}`，worker 跑完异步 push `task_complete`。「同步死等」纯是**客户端选择在 ack 后继续堵 `recv()`**。所以异步化**不需要任何上传协议开关**：客户端收到既有 ack 后转去轮询，服务端无需知情。`wait` 开关 + `task_accepted` 是多余 churn，删掉。

## 1. 背景与根因

批量 100+ 任务报 `任务队列已满` + `接收消息超时(300s)`。止血修了内存/资源泄漏 + 队列满优雅信号，但**修不掉 300s 超时**：pool=1（真实并发=1）时位置 k 的任务约 `k×T` 完成，客户端 300s 窗只容 `k ≤ 300/T` 个，深队列后段**注定**等不到，`max_queue_size` 设多大都救不了。

| 队列深度 | 当前同步模式 | 异步轮询模式 |
|---|---|---|
| 1~3 轻交互 | ✅ 正常 | ✅ 正常 |
| ~5（T=60s 临界） | ⚠️ 临界超时 | ✅ 正常 |
| 10~20 中等批量 | ❌ 第 6 位起几乎全超时，误判「服务挂了」 | ✅ 全部最终拿到，进度可见 |
| 100+ 重批量（事故现场） | ❌ 95%+ 超时 | ✅ 漫长但不丢不误判 |

异步**不提吞吐**（总耗时仍 `~100×T`），解决的是「客户端不再因排队误判超时挂掉」。

## 3. 协议规格（上传路径不变，仅客户端行为变 + 加 batch 查询）

### 3.1 客户端要处理的消息全集（按 wire shape，非臆想）

| 消息 | wire | 何时 | client 动作 |
|---|---|---|---|
| `task_complete` | `type:"task_complete"` | 缓存命中（单文件 upload_request 阶段 / 分片）；或 worker 跑完 push | **任何时刻收到即终结**该 task |
| `task_queued` / `upload_complete` | `type:"task_queued"`/`"upload_complete"`，带 `task_id`(canonical)/position/estimated_wait/message | 入队成功 | **不堵等**，转入轮询循环 |
| `queue_full` | `type:"queue_full"` | 队列满 | 退避，发 `finalize_upload`（分片不重传）重试 |
| `error{task_expired/task_not_found}` | `type:"error"`，`data.error` | 轮询查不到 | 凭 file_hash 重投（见 §5.4） |
| `task_status` / `task_status_batch` 响应 | 见 3.2 | 轮询应答 | 按 status 推进；含终态 |

### 3.2 异步轮询时序（无新上传消息）

```
client                                   server
  │ upload_request / chunk×N / finalize ─▶ 命中→task_complete{result}
  │                                         未命中入队→task_queued{task_id,position,est}
  │ ◀── task_queued{canonical task_id} ──  (今天就有, 不变)
  │ … client 不堵等, 转轮询 …
  │ task_status_batch{task_ids:[…]} ─────▶ 逐 id: status/progress(+完成项内联 result)
  │ ◀── [{id, status:processing},          (PROCESSING→空result小帧;
  │      {id, status:completed, result}, …] completed/failed/timed_out→内联 result/error)
  │ (某 id 终态即从轮询集移除)
```

### 3.3 新增消息：`task_status_batch`（真·新代码）

- 请求 `{ "type":"task_status_batch", "task_ids":[...] }`，上限 `task_ids ≤ 50`（截断 + warn，控帧大小）。
- 响应**逐 id** 项含 `{task_id, status, progress, result?}`：
  - PROCESSING/PENDING → `result:null`（小帧）。
  - **COMPLETED → 内联 result**（JSON 走 `TranscriptionResult`；**SRT 走 `srt_content`**，codex #11：`TaskStatusResponse.result` 装不下 SRT）。
  - **FAILED/TIMED_OUT/CANCELLED → 终态 + error**（codex #10：client 不能只盯 completed，否则失败任务无限轮询）。
  - expired/not_found → 该 id 标 `expired`/`not_found`（不整批失败）。
- **不带 live queue_position**（codex #12：`asyncio.Queue` 无 task_id 位置 API；position 只在入队 ack 给一次快照，够首估）。
- 服务端复用 `get_task`+`was_evicted` 循环 + 单查的 SRT/JSON result 组装逻辑。

### 3.4 新增 schema

`TaskStatusBatchResponse`（list of per-id item，item 含可选 result/srt_content/error）。**`FileUploadRequest` 不加任何字段**（无 `wait`）。

## 4. 服务端改动清单

1. **`src/models/schemas.py`**：加 `TaskStatusBatchResponse` + per-id item model（status/progress/result?/srt_content?/error?）。**不改 `FileUploadRequest`**（无 `wait`）。
2. **`src/api/websocket_handler.py`**：加 `_handle_task_status_batch` + `_handle_message` 路由 `task_status_batch`；逐 id **复用 `_send_task_status` 的 result/SRT/expired 组装形态**（DRY，不要第 4 次复制 SRT/JSON shaping）；`task_ids` 上限 50 截断 + warn。

**落地约束（/plan-eng-review）**：
- batch handler 读 `task.status` + `task.result` 必须在**同一同步块、中间不 await**——否则撞 `task_manager.py:499-500` 注释的「COMPLETED 翻转在 result 组装后」窗口，会返回 status=completed 但 result=null。`_send_task_status` 已安全，照抄。
- 终态全集：completed/**failed/timed_out/cancelled** 都逐 id 返终态（含 error），client 据此停轮询。
- 内联 result 上限 50/批：长音频 JSON 50 份单帧可能数 MB，靠「client 把已完成 id 移出轮询集，每 result 只发一次」摊平；50 是起步硬上限，实测帧过大再降。

**不改**：worker 主体、database 折维、引擎路径、**上传消息协议**、task_manager（本轮 batch 纯读，零写穿）。

⚠️ 改 schemas + websocket_handler（task_manager 不动）→ **必跑 FunASR parity**：
`FUNASR_RUN_INTEGRATION=1 FUNASR_DEFAULT_ENGINE=funasr venv/bin/python -m pytest tests/integration/test_parity_funasr_semantic.py`

## 5. 客户端契约（两个客户端各自实现，无新上传字段）

1. 上传后**不堵等 `task_complete`**：收到 `task_queued`/`upload_complete` ack（带 canonical task_id + position）即转轮询循环。
2. **任何时刻收到 `task_complete` 即终结**该 task（缓存命中可能在 finalize 前就来；worker push 在 client 仍连时也会到，可当快路径）。
3. **批量轮询硬约束**：批量场景**必须**用单条 `task_status_batch`（上限 50/批，>50 分批），**禁止 per-task 各自轮询**；用 ack 的 `estimated_wait` 定首次延迟；间隔退避（如 5~10s）。完成项 result **随 batch 内联拿到**，不再单查（codex #13）。
4. **终态全集**：轮询要识别 `completed/failed/timed_out/cancelled` 四个终态都停轮询（codex #10），不能只认 completed。
5. **poll-miss 兜底**：`error` 里 `task_expired` **或 `task_not_found`** 都凭 file_hash 重投（重启令所有轮询 not_found，因 `was_evicted` 也在内存）。已完成重投命中 DB 缓存秒回；**未完成/重启后重投**：分片 `finalized_file_path` 在内存 session，**重启即失效 → 必须重传**（止血的「不重传」仅 server 未重启的 queue_full 重试成立）。
6. **以 ack 的 task_id 为准**轮询（前向兼容：未来 in-flight 去重 PR 上线后，ack 的 task_id 可能是折叠后的 canonical id）。

**附带收益**：异步 client 断连重连后仍能凭 task_id 轮询拿结果（同步模式断连即丢 push）。

## 6. 边界与错误地图

| 场景 | 处理 | 状态 |
|---|---|---|
| 轮询时 PENDING/PROCESSING | 非终态永不清→必命中；result=null 小帧 | ✅ |
| 终态 failed/timed_out/cancelled | batch 内联终态 + error，client 停轮询 | 新代码覆盖 |
| 断连重连后轮询 | TTL 内命中 self.tasks | ✅ 白赚 |
| 完成时 client 离线 | self.tasks(TTL 1h) + DB 缓存 | ✅ |
| 轮询超 TTL | `task_expired` → 重投命中 DB 缓存 | ✅ |
| 重启（队列/在途全丢） | 全 `task_not_found` → 重投（分片需重传）；完成的命中 DB 缓存 | ⚠️ client 兜底 + 重传代价 |
| 双客户端并发同一文件 | **本轮不去重**：第二次命中第一次缓存或幂等覆写，仅白跑一个 pool 槽（无正确性 bug）；去重见独立 PR | 推迟（独立 PR） |
| SRT 结果轮询 | batch/单查走 srt_content 非 result 字段 | 新代码覆盖（codex #11） |
| batch 100 同轮完成 | batch 内联 result，无单查风暴 | ✅（codex #13） |
| batch task_ids 超 50 | 截断 + warn | 新代码覆盖 |

**无 RESCUED=N + Silent 的 CRITICAL GAP**；重启重传是已显式声明的已知代价。

## 9. NOT in scope

- **`wait` 开关 / `task_accepted` 新消息**：codex 证明多余——客户端在既有 ack 后轮询即可，服务端无需知情。

- **持久任务表（方案 B）**：重启存活由 file_hash 缓存 + 重投兜底，不值新表 + 迁移 + 写穿 + parity 风险。多 worker/多机再触发。
- **确定性 task_id**：随机 uuid + 重投已能恢复；与 force_refresh 纠缠。
- **per-连接限流**：→ TODOS。
- **吞吐扩容（pool>1）**：正交，受显存约束，另立。
- **重启后在途任务计算续跑**：worker await 无法持久化，B 也做不到；重启=重投重跑，接受。
- **batch live queue_position**：`asyncio.Queue` 无位置 API，入队 ack 一次性快照够用。
