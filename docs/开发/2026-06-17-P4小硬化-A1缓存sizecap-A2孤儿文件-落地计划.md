## 背景 / 问题

两个运营缺口,都是"无限增长 / 资源泄漏"型兜底缺失:

1. **DB 缓存只有 TTL 无数量上限**:`clean_old_cache()`(`database.py:482`,由 `main.py`
   `_periodic_cleanup` 每小时调)只按 30 天 TTL 删,30 天内若高频转录可堆任意多条。
   `task_manager` 的 `_evict_terminal_tasks` 早有 "TTL + size-cap 双保险" 范式,缓存表缺对称兜底。

2. **孤儿音频文件**:任务完成/失败/取消三处会删上传文件(`delete_after_transcription` on +
   无同 hash live 任务)。但**看门狗** `_terminalize_stale_processing` 把卡死任务标
   `TIMED_OUT` **只改状态不删文件**;这类(及任何删文件失败的)终态任务被
   `_evict_terminal_tasks` 淘汰时文件无人回收 → 孤儿堆在 upload_dir。

## A1:DB 缓存 size-cap

**设计**:照 `_evict_terminal_tasks` 的 "TTL 先清,再 size-cap 挤" 范式,给 `clean_old_cache()`
加第二阶段。

- `DatabaseConfig` 加字段 `max_cache_count: int = 5000`(数量硬上限,命名对称现有 `max_cache_days`)。
  - `<= 0` ⇒ 不限制(关 size-cap),防误配清空缓存。
- `_apply_env_overrides` 加 `FUNASR_MAX_CACHE_COUNT` 覆盖(走标准 `_override_if_set`,`print_config` 可见)。
- **(eng-review F3 定案 A)** 加索引 `idx_accessed_at ON transcription_cache(accessed_at)` —— 现有
  只有 `idx_created_at`,size-cap 用 `accessed_at` 排序若无索引会全表扫+排序。索引建在 schema 初始化处
  (`CREATE INDEX IF NOT EXISTS`,迁移安全)。
- `clean_old_cache()` 在现有 TTL `DELETE` 之后:
  ```sql
  -- 仅当 max_cache_count > 0
  DELETE FROM transcription_cache WHERE id IN (
    SELECT id FROM transcription_cache
    ORDER BY accessed_at ASC, id ASC  -- true LRU + 确定性 tie-breaker(秒级精度突发会 tie)
    LIMIT <overflow>                  -- overflow = COUNT(*) - max_cache_count
  )
  ```
  先 `SELECT COUNT(*)` 算 overflow,>0 才删。**(F3 定案 A)** LRU 用 `accessed_at`(高频复用的转录存活),
  非 `created_at`——后者只挤"创建最老"会误伤仍高频复用的旧缓存。**(codex)** 加 `id ASC` tie-breaker 防
  同秒 accessed_at 顺序不确定。
- **软上限非硬 cap**:`clean_old_cache` 每小时跑,count 可在小时内短暂超 `max_cache_count`,这是兜底防
  无限增长,非实时硬限(可接受)。
- **(codex)顺手修既有 regression**:`clean_old_cache` 的 TTL 比较有时间戳格式 bug——DB 存
  `CURRENT_TIMESTAMP`(空格 `2026-06-17 12:00:00`)但 cutoff 用 Python `isoformat()`(`T` 分隔),
  字符串比较 `'T'(0x54) > ' '(0x20)` 会误删临界同日行。改 cutoff 用
  `strftime("%Y-%m-%d %H:%M:%S")`(空格,对齐 DB 格式)。**IRON RULE 加回归测试钉死格式**。
- 日志:删了 N 条按 size-cap(与 TTL 删分别记数,便于排查)。
- **VACUUM 不做**:删行不缩 DB 文件(freelist 复用),但 5000 小行规模 freelist 复用足够,不引入 vacuum 策略。

**TDD**:`tests/unit/test_database_cache_sizecap.py`
- 插 max+k 条(不同 accessed_at)→ `clean_old_cache()` → 断言剩 max 条 + 留下的是 accessed_at 最新那批。
- `max_cache_count<=0` → 不删任何(只走 TTL)。
- 未超上限 → size-cap 阶段不动。

## A2:孤儿文件清理(F2 + 跨模型定案:文件系统 sweeper)

**孤儿来源**:看门狗 `_terminalize_stale_processing` 标 `TIMED_OUT` 时**只改状态不删文件**;
这类(及删失败 / 进程重启前未删的终态)文件无人回收。

**为什么不用 evict 兜底(codex 二审推翻 F2-A)**:evict 挂清理有三洞——① 看门狗不杀 worker,
evict(1-2h 后)删文件可能撞**还在读的 worker**;② **进程重启后 `self.tasks` 没了,孤儿永久不可见**;
③ 删失败被 `delete_file` 吞掉无重试。**持久化边界错位**:孤儿是文件系统事实,不该锚在内存 task 生命周期。

**定案:upload-dir 文件系统 sweeper**(同构现有 `_sweep_upload_sessions` / `cleanup_temp_files`):

```python
async def _sweep_orphan_upload_files(self) -> int:
    """扫 upload_dir, 删 mtime 超宽限期 且 无 live 引用 的孤儿文件。

    live 引用 = 任一 PENDING/PROCESSING 任务的 file_path, 或任一 upload session 的
    finalized_file_path。宽限期 orphan_file_grace_seconds 必须 >> task_max_processing
    (确保 worker 早读完), 规避 unlink-under-worker。扛进程重启(文件系统为准)+
    删失败下轮重试。返回删除数。
    """
```

- 接入 `_maintenance_loop`(已在 `_queue_lock` 下周期跑;sweeper 读 upload_dir + 查 self.tasks/
  upload_sessions 引用,纯本地)。也可挂 `main.py._periodic_cleanup`,但 `_maintenance_loop`
  能直接拿 task 引用,更内聚。
- **跨进程引用**:upload session 在 `websocket_handler`,task 在 `task_manager`。sweeper 需同时查两者
  的引用集——通过 `ws_handler` 单例拿 `upload_sessions` 的 `finalized_file_path`(import 现成单例,
  照 task_manager 既有 `from src.api.websocket_handler import ws_handler` 用法)。
- **宽限期**`orphan_file_grace_seconds`(`TranscriptionConfig`,默认 `7200`=2h,env
  `FUNASR_ORPHAN_FILE_GRACE_SECONDS`)。必须 > `task_max_processing_seconds`(1h)+ 裕量。
- **删除诚实**:`delete_file` 改返回 `bool`(删成功/文件不存在的区分),sweeper 据此记真实回收数。
- **不动 `_evict_terminal_tasks` 返回契约**(保持 int,既有调用/测试不破)。

**TDD**:`tests/unit/test_task_orphan_sweep.py`
- upload_dir 放一个 mtime 超宽限期、无 live 引用的文件 → sweep → 删除 + 返回 1。
- 文件被 PENDING/PROCESSING 任务 file_path 引用 → **不删**。
- 文件被 upload session finalized_file_path 引用 → **不删**(mock ws_handler.upload_sessions)。
- mtime 在宽限期内(新文件)→ **不删**(防删在途上传)。
- 空 upload_dir / 目录不存在 → 不报错返回 0。

## 风险 / 边界

- **A1**:`accessed_at` 须确认所有读路径都更新(已核 database.py:364 主路径更新;投影/跨引擎回退路径
  若不更新,这些行 accessed_at 偏旧可能被优先挤——可接受,本就是冷数据)。size-cap 是软上限兜底,
  默认 5000 宽松,不激进。
- **A2 sweeper**:宽限期 `> task_max_processing_seconds` 保证 worker 早读完再删(规避 unlink-under-worker)。
  扛进程重启 + 删失败下轮重试。绝不删 live 任务/session 引用的文件。`delete_after_transcription=False`
  时——sweeper 是兜底磁盘保护,与该开关正交;但为尊重"用户要留文件"语义,sweeper 在
  `delete_after_transcription=False` 时**整体不跑**(与现役删除语义一致)。
- 两块都改核心路径(database.py / task_manager.py)→ 落地后跑 `bash scripts/run_checks.sh --parity`。
