# 高负载队列机制 — 止血修复计划

> 来源：`/plan-eng-review`（2026-06-16）+ codex 外部声音
> 触发：Mac 生产环境批量提交 100+ 任务报 `任务队列已满，最大容量: 100` + 客户端 `接收消息超时(300秒)`
> 方向：**C — 先止血，异步轮询契约作为下一步**（见 TODOS.md #20）
>
> **✅ 已落地（2026-06-16，3 个 commit）**：共享契约（QueueFullError + 6 config 字段 + TIMED_OUT）→ Lane A（task_manager 内存清理 + 看门狗 + queue-full 回滚 + EMA）→ Lane B（finalize 资源处理 + queue_full 信号 + finalize_upload 幂等 + session sweep + expired/not_found）。新增 37 单测全绿；FunASR parity + smoke wiring 9 passed 无回归。实现要点见 CLAUDE.md「任务队列与高负载健壮性」节。

## 一、根因诊断（三个症状，根因不同）

```
客户端 (单 WebSocket, 同步等 task_complete, 300s 超时)
   │ 批量提交 100+ 文件
   ▼
分片上传 → _finalize_chunked_upload (websocket_handler.py:670-673)
   │ create_task → self.tasks[id] = task        ← ① 永不删除 (内存无限增长)
   │                                               ↳ 且在容量检查"之前"就插入 (codex 窟窿)
   ▼
submit_task (task_manager.py:178-205)
   │ async with _queue_lock:
   │   qsize() >= 100 ? ──► raise "队列已满"      ← ② 第101个直接拒 (泛化 finalize_error)
   │                          │ except 只 send_error，不清理
   │                          ▼
   │                       session + temp_file 泄漏  ← ③ 失败路径资源泄漏
   │ put_nowait(id)
   ▼
task_queue (asyncio.Queue, maxsize=100)
   │
   ▼ 2 worker (max_concurrent_tasks=2)，each → qwen3 pool (pool_size 默认 1)
   │   有效并发 ≈ 1，每任务数十秒~分钟级
   ▼
队列位置 N 的任务等待 ≈ N × 单任务时长 >> 300s  ← ④ 客户端必然超时
```

| 症状 | 根因 | 本轮能否修 |
|---|---|---|
| `队列已满` | 队列健壮性（硬拒 + 内存泄漏） | ✅ 本轮 |
| 内存撑爆（静默 OOM） | `self.tasks` 无清理 | ✅ 本轮 |
| `接收消息超时(300s)` | 客户端同步死等 + pool=1 深队列 | ❌ 需异步轮询（#20） |

**关键认知：本轮止血让"队列满"优雅、让内存不爆，但修不掉 300s 超时**——客户端只要还同步死等，深队列就还超时。300s 的真正根治是 #20 异步轮询。

## 二、本轮修复 scope（止血 + codex 加固）

### 1. self.tasks 内存清理（决策 C：TTL + size-cap 双保险）
- **TTL**：终态任务（COMPLETED/FAILED/CANCELLED/TIMED_OUT）超 TTL（默认 ~1h，config 字段）周期清理。TTL 必须 ≥ 客户端轮询窗口。
- **size-cap**：硬数量上限（默认 ~500，config 字段），超了从最老终态任务挤。内存上界确定。
- **非终态不清**，但配 ↓ 看门狗防卡死。
- **🔴 queue-full 回滚（codex 必修）**：`create_task` 在容量检查前已写入 `self.tasks`，队列满时必须回滚该插入（或把容量检查提到 `create_task` 之前），否则被拒的 PENDING 任务永久泄漏。
- **processing 看门狗（决策 A）**：按 `started_at` 扫描，超过 `max_processing_time`（单任务上限的若干倍，config 字段）仍在 PROCESSING 的 → 强制标 `TIMED_OUT` → 进终态被正常回收 + 释放并发名额。闭合"非终态永不清"的洞（ASR 卡死 / worker 挂 / 取消路径坏）。

### 2. 队列上限（决策 A：压低 + 可配置）
- 默认压到"超时窗可消化"（≈ `300s ÷ 平均单任务时长`，个位数），做成 config 字段。
- 被接受 = 大概率超时前做完（诚实）；其余 `queue_full` 退避。
- 异步轮询（#20）上线后用 env 调大。

### 3. queue-full 结构化信号（决策 A：429 语义）
- 独立 `queue_full` 消息类型 + `retry_after` 秒数（按 `队列位置 × EMA均值`，见 ↓5）。
- **向后兼容（codex）**：同时带老客户端能识别的 error code/shape + 结构化字段，避免新类型被旧客户端忽略。
- 保留队列上限作为准入控制。

### 4. finalize 资源处理（决策 A：保留供重试 + TTL，真错才清）
- **queue-full**：保留 session + temp 文件，客户端重发 finalize 不重传大文件。
- **幂等（codex）**：session 加 `submitted` 状态，重发 finalize 时已提交 → 直接回 task_id，不二次入队（防 double-submit）。
- **session TTL + 硬上限（codex）**：扫遗弃 session 清 temp 文件；TTL 之外加硬数量/磁盘容量上限 + 主动 sweeper，防恶意/坏客户端堆垃圾。
- **sweep 与 finalize 互斥（codex）**：per-session 状态/锁，避免 sweep 删文件时撞上 finalize 重试。
- **错误分级（codex）**：**提交前**错误（save 文件后、入队失败）→ 清文件；**提交后**错误（如 ws 发送响应失败）→ **绝不删**文件，任务要用它。
- **状态区分（codex）**：被清理任务轮询返回 `expired`，从未存在返回 `not_found`。

### 5. estimated_wait（决策：实际 RTF）
- 改 `队列位置 × 平均单任务时长`，均值用 **EMA / 近期中位数**防长音频污染，加冷启动默认值。
- 替换 task_manager.py:194 硬编码的 2 分钟。

### 6. TODO #20 异步轮询（已记 TODOS.md，补约束）
- 补：`self.tasks` 内存态，**重启丢全部状态**。异步轮询若让客户端依赖它，最终要么持久化任务状态，要么明确"状态跨重启易失"语义。

## 三、测试（全套 GAP，按项目 TDD 铁律全写，红→绿→commit）

```
[+] task_manager 内存清理
  ├── 终态超 TTL → 清理 ; size-cap 超上限 → 挤最老终态
  ├── ★★★ PENDING/PROCESSING 永不被 TTL/cap 清
  ├── ★★★ queue-full 回滚 self.tasks 插入 (codex 窟窿回归测试)
  ├── ★★★ processing 看门狗：超 max_processing_time → TIMED_OUT → 可回收
  └── ★★★ 清理后 get_task→None, _send_task_status 返回 expired (回归)
[+] queue-full 结构化信号
  ├── 队列满抛专用异常类型 (非泛化 Exception)
  └── retry_after = 队列位置 × EMA均值
[+] finalize 资源处理
  ├── ★★★ queue-full → queue_full 消息, session+文件保留
  ├── ★★★ 重发 finalize 幂等 (已提交→回 task_id 不二次入队)
  ├── 提交前错误 → 清文件 ; 提交后错误(ws send 失败) → 不删文件
  ├── session TTL/硬上限 sweeper → 清遗弃 ; sweep 与 finalize 互斥
  └── 非分片上传路径(:290) 同样映射 queue_full
[+] estimated_wait：基于真实均值 + 冷启动默认
[~] [→E2E] 批量提交 >上限 → 部分 queue_full → 客户端退避重投 → 最终全完成（延后，见下）
[~] [→E2E] 完成任务 TTL 窗内轮询命中, 超窗走 DB 缓存兜底（延后，见下）
```

**已落地：37 个 unit 全绿**（test_queue_resilience_contract.py 14 + test_task_manager_queue_resilience.py 14 + test_websocket_finalize_resilience.py 9），含 4 个 ★★★ 关键不变量/回归（queue-full 回滚、非终态永不清、看门狗→TIMED_OUT、expired vs not_found）。FunASR parity + smoke wiring 9 passed 无回归。

**延后（integration tier）**：2 条 [→E2E]（真 subprocess server + ws client + 真模型填队列）heavy/flaky，单测已覆盖全部逻辑不变量，E2E 仅验证集成接线。建议随 TODOS #20 异步轮询一起补（届时 ws 轮询契约也要 E2E）。

## 五、NOT in scope（本轮明确不做）

- **异步轮询契约重构** → TODO #20（P1，300s 超时真正根治，需改客户端 + 部署协调）。
- **任务状态持久化 / 跨重启存活** → 随 #20 一起（本轮 self.tasks 仍内存态）。
- **提升吞吐**（pool_size、GPU 并发）→ 与队列健壮性无关，真瓶颈但非本主题。
- **per-用户/连接限流配额** → 若"100+ 批量是否合理需求"成为产品议题，单独走 /plan-ceo-review。
- **WebSocket 重连/会话恢复** → 客户端侧，超本轮范围。
