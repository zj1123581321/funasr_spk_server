## 1. 背景：刚做完的「止血」（已上生产）

起因：Mac 生产环境批量提交 100+ 任务，报 `任务队列已满，最大容量: 100` + 客户端 `接收消息超时(300秒)`。经 `/plan-eng-review` + codex 评审，诊断出**三个根因叠加**，本轮走「止血（决策 C）」，已落地上生产（commit 5f7eabf→9b7eec7，prod 已 PM2 重启验收）。

**止血修了什么**（详见 `docs/开发/2026-06-16-高负载队列机制-止血修复计划.md` + CLAUDE.md「任务队列与高负载健壮性」节）：
- 队列满 → 结构化可重试 `QueueFullError`/`queue_full` 信号（429 语义），`max_queue_size` 现网设 150
- `self.tasks` 内存无限增长 → TTL（默认 1h）+ size-cap（默认 500）双保险清理，非终态永不清
- codex 窟窿：`create_task` 先写 `self.tasks`、`submit_task` 才查容量，队列满未回滚 → 已修（`self.tasks.pop` 回滚）
- 卡死 PROCESSING → 看门狗强制 `TIMED_OUT`（默认 1h）
- finalize 失败资源泄漏 → 错误分级 + session TTL/上限 sweep + `finalize_upload` 幂等重试（不重传大文件）
- 轮询 `task_expired` vs `task_not_found` 区分

**止血明确没修的（本次 session 的核心议题）**：
- ❌ `接收消息超时(300s)` 的**根因** = 客户端单 ws 同步死等 + pool=1 深队列。无论队列上限设多少，pool=1 下排在深处的任务等待 ≈ 位置 × 单任务时长 >> 300s，**必然超时**。
- ❌ 现网客户端**还不认识 `queue_full` 新消息**（当成 error，不会自动退避重投）。

## 2. 当前任务系统架构（讨论的起点）

```
客户端(单 WebSocket, 同步等 task_complete, 300s 超时)
   │ 1. upload_request / chunked → 2. upload_data/chunk → 3. (自动)finalize
   ▼
WebSocketHandler (src/api/websocket_handler.py)
   │ create_task → self.tasks[id]=task (进程内 dict, 易失)
   │ submit_task → task_queue (asyncio.Queue, 进程内, 易失)
   ▼
TaskManager (src/core/task_manager.py)
   │ N=max_concurrent_tasks 个 worker 从 queue.get()
   │ 每 worker → resolve_transcriber(engine).transcribe()
   │   qwen3 → Qwen3InProcPool/Qwen3PoolTranscriber (pool_size, 默认 1)
   │   ⇒ 真实并发 ≈ pool_size，worker 阻塞在池上
   ▼
结果双写：self.tasks[id].result (内存) + db_manager.save_result (DB 缓存, 按 file_hash+engine)
   │
   ▼ 推送 task_complete 给当前 ws  ←── 客户端在这里同步等
轮询接口已存在但批量路径没用：task_status 消息 → _send_task_status → get_task(self.tasks)
```

**关键事实**：
- 真实并发 = `qwen3_pool_size`（Mac prod profile 默认 1），不是 `max_concurrent_tasks`。
- 结果**已经双写 DB 缓存**（按 `(file_hash, engine+折维tag)`），所以「结果落地」其实不依赖内存。
- `task_status` 轮询接口**已存在**，但读的是 `self.tasks`（内存、重启即丢）。
- 项目长音频路径**早已因 ws 断连改用轮询**（CUDA box 实践），轮询是本项目已验证可行的成熟模式。

## 3. 整个任务系统的结构性设计问题（讨论清单）

1. **同步请求-响应模型 vs 任务时长不匹配**：ws 同步等 300s，但单任务可能数十秒~分钟，pool=1 深队列必超时。这是核心矛盾。
2. **状态易失**：`self.tasks`（任务状态）和 `task_queue`（待处理队列）都在进程内存。**服务重启 → 所有进行中/排队任务状态丢失**；已 submit 未处理的任务直接蒸发。轮询依赖 `self.tasks`，重启后轮询全 miss。
3. **结果可达 vs 内存清理的张力**：止血加了 TTL/size-cap 清理，但 `task_status` 读内存。结果其实在 DB 缓存里（按 file_hash），**轮询若改读 DB 可同时解决「重启存活」+「清理后仍可取」**。值得讨论。
4. **客户端无背压处理**：`queue_full` 现网客户端不认，触发即用户可见失败。客户端契约要同步演进。
5. **吞吐天花板**：pool=1 是真瓶颈。队列改动不提升吞吐。是否要谈 pool 扩容/排队预期管理/限流，而非假装队列能消化。
6. **无 per-用户/连接限流**：一个客户端能甩 100+ 把队列占满（TODOS #15 num_speakers per-request 闲置；限流也缺）。
7. **无重连/会话恢复**：ws 断了，进行中任务的结果送不回；客户端只能靠轮询/缓存重提兜底。
8. **看门狗治标**：只能把卡死任务状态终态化，**不能真杀在途 worker 的 await**——真卡死的 worker 槽位回收需 worker 级取消，止血范围外。

## 4. 异步轮询契约（TODOS #20）规划要点

目标：把批量/分片路径从「finalize 即同步等 `task_complete`」改成「finalize 立刻返回 `task_id` → 客户端轮询 `task_status`」，根治 300s 超时。

待拍板的设计维度：
- **协议形态**：纯轮询？还是「ws 在线就推 task_complete + 离线/超时可轮询」双模？双模兼容性最好但更复杂。
- **状态持久化**：是否把任务状态落 DB（sqlite，项目已用），让轮询读 DB 而非内存 → 同时解决重启存活 + 清理后可取。这是 §3.2/3.3 的关键解药，重点讨论。
- **结果获取**：轮询返回状态；done 后结果从哪取（内存 / DB 缓存按 file_hash 命中，已现成）。
- **TTL 与轮询窗口耦合**：止血的 `task_retention_ttl_seconds`（1h）必须 ≥ 客户端最长轮询窗口；若状态落 DB 则约束转移到 DB 保留期。
- **向后兼容 + 部署顺序**：老同步客户端必须仍能用（等 task_complete）。**先服务端、后客户端**（与 diarize 上线同款假设，需单测钉死）。
- **客户端改造**：现网客户端（CapsWriter 风格）要改成提交后轮询 + 处理 `queue_full` 退避重投。评估改造量。
- **限流/准入**：异步化后是否顺带加 per-连接并发上限（避免单客户端霸占队列）。
