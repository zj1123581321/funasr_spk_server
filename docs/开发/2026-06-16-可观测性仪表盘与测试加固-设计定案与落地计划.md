## 4. P1 详细设计

### 4.1 集成点（同端口 process_request）

`src/main.py` 当前 `websockets.serve(handler, host, port, ...)`（约 :56-69）。`websockets` 的 `serve()` 接受 `process_request` 回调：在 WebSocket 握手前拦截每个 HTTP 请求，**若返回 `(status, headers, body)` 则直接以普通 HTTP 响应返回，不进入 ws 升级**；返回 `None` 则照常走 ws 握手。

```
                       ┌─ path 命中 /health,/metrics,/ ─► 返回 HTTP 响应 (不升级)
HTTP 请求 ─► process_request ─┤
                       └─ 其它 (ws 升级请求) ─► return None ─► 走现有 websocket handler
```

新增模块：`src/api/http_endpoints.py`
- `async def http_process_request(path, request_headers) -> Optional[tuple[HTTPStatus, Headers, bytes]]`（**12.0 legacy 签名**，codex #3）
  - **只处理 GET**（codex #1：HEAD/POST/带 body 的请求到不了钩子干净返回；探针若用 `HEAD /health` 会假失败——文档化要求探针用 GET）
  - `path` **含 query string**（codex #2）：必须 `urllib.parse.urlsplit(path)` 取 path 部分做路由，query 部分取 token；**禁止**直接 `path == "/metrics"` 等值比较（否则 `/metrics?token=x` 漏路由 + 断鉴权）
  - 命中 `/health` `/metrics`（`/` 见 NOT in scope）则组装响应；否则 `return None` 走 ws 升级
  - 注入依赖：`task_manager`、`db_manager`、`config`（构造闭包或轻量类持有引用）
- 纯函数指标采集器集中在此模块，便于单测

⚠️ **`websockets==12.0` legacy 签名（已核实，A2）**：本项目锁 `websockets==12.0`，用 **legacy `serve()` API**，回调签名是 `async def process_request(path, request_headers) -> Optional[tuple[HTTPStatus, Headers, bytes]]`（不是 13+ 的 `(connection, request)` + `connection.respond()`，那是 asyncio 新 API，12.0 没有）。返回元组则直接 HTTP 响应，返回 `None` 则照常 ws 升级。
- **耦合警告**：健康端点把我们钉在 websockets legacy API 上。将来 bump 到 13+ 这套 API 变/废，会破。落地必须写一个**钉版本行为的单测**（断言 process_request 在 12.0 签名下的返回契约），让盲目升级直接红，而非运行时静默坏。

### 4.2 指标清单（全部复用现有内部状态）

| 指标 | 数据来源 | 备注 |
|---|---|---|
| `funasr_queue_size` / `_max` | `task_manager.task_queue.qsize()` / `max_queue_size` | 准入控制可见 |
| `funasr_tasks_inflight` | PROCESSING 状态计数 | 真实并发 |
| `funasr_tasks_pending` | PENDING 状态计数 | 等待深度 |
| `funasr_task_seconds_ema` | `task_manager._estimate_task_seconds()` | EMA 处理时长 |
| `funasr_tasks_terminal_total{status=}` | **终态化点累加的单调 Counter**（completed/failed/timed_out/cancelled）；**不是扫 self.tasks**（A1） | 错误率分子 |
| `funasr_tasks_evicted_total` | TTL/size-cap 淘汰计数 | 内存清理可见 |
| `funasr_tasks_stale_timeout_total` | 看门狗 TIMED_OUT 计数 | 卡死可见 |
| `funasr_cache_hits/_misses` | `db_manager.get_cache_stats()` | 命中率 |
| `funasr_cache_projected_serves` | `get_cache_stats()` 的 `projected_serves` | diarize 投影命中 |
| `funasr_cache_rows` / `_db_bytes` | DB 行数 / 文件体积 | 增长可见（喂 P4 size-cap） |
| `funasr_engine_info{engine,pool_size}` | `config` + 池 | 哪个引擎在跑 |
| ~~`funasr_pool_busy/_idle`~~ | **砍掉（A4）**：FunASR 多进程池（Mac 默认引擎）无忙闲可读，Qwen3 只能读 `_available.qsize()`。改用上面 `funasr_tasks_inflight`（PROCESSING 计数）当通用在途 gauge，引擎无关、数据现成 | 已合并进 inflight |
| `funasr_vram_free_mib` / `_used_mib` | `gpu_mem` 探针，**仅 cuda**，mac 省略 | ⚠️**P1 codex #11**：`gpu_mem` 用同步 `subprocess.run(timeout=5)`，**禁止在 process_request 里直接调**（冻事件循环最多 10s/scrape，所有 ws 处理停摆）。改：后台协程按 TTL（如 10s）刷一个缓存值，端点只读缓存；探不到/mac 则不出该指标 |
| `funasr_uptime_seconds` / `funasr_build_info{version}` | 启动时间戳 / 版本 | meta |
| `funasr_errors_total{kind=}` | **需新增轻量计数器** | 现错误只散在日志，见下 |

**单调计数器埋点（A1，核心修正）**：`tasks_terminal_total` 和 `errors_total` **都必须是终态化/catch 点累加的进程级单调 Counter**，**绝不能扫 `self.tasks` 现算**——`self.tasks` 被 TTL 淘汰（终态保留 ~1h 即清），扫描值是"当前驻留 gauge"不是"累计 total"，Prometheus `rate()`/错误率告警会在任务淘汰后**静默回退**（踩中零静默失败红线）。落地：
- `errors_total{kind}` 在 `task_manager.py:475-822` / `websocket_handler` catch 点累加，按 kind 分（queue_full / engine_error / timeout / oom_fallback / ...）。**已核实**：worker 错误确实传回主进程这些 catch 点，主进程计数器埋点安全。
- `tasks_terminal_total{status}` **覆盖全部终态化点**（codex #14：不止一处！`submit_task` 缓存命中、`upload_request` 缓存命中、正常处理完成、重试失败、取消、看门狗超时）——取代 `task_manager.py:829-834` 的遍历现算。落地建议：抽一个 `_record_terminal(task, status)` 单点，所有终态化路径都走它累加 + 写状态，杜绝漏点（make-the-change-easy：先重构出单点，再加计数）。
- `errors_total{kind}` **边界明确**（codex #13）：除了 catch 到的异常，client 可见的 early-return 软错误（格式非法 / 文件超大 / hash 坏 / task_not_found / 鉴权失败）也要显式累加；落地时列全 kind 清单，钉测试。
- 其余指标（queue_size / inflight / pending / cache）是**瞬时 gauge**，读现有状态即可（`self.tasks` 只在主事件循环协程里读写、无锁但协程原子，process_request 同循环读安全）。

### 4.3 /health 状态机（不是简单 200）

**A3 终定（codex 纠正后）：/health 只做 liveness，保持简单。**

⚠️ **前提纠正**：模型是**启动时 eager 加载**（`src/main.py:47-49` `transcriber.initialize()` 在 `:56 websockets.serve()` 之前），**不是懒加载**。所以"能接但没热"的窗口不存在——等 HTTP 端点能应答时模型早已加载完，`model_warm` 几乎恒真。原 A3 的懒加载前提作废。

```
liveness 检查项 (全 AND，决定 200/503):
  ① 维护循环在跑   (task_manager._maintenance_task is not None and not done())
  ② 能接任务       (TaskManager.is_running)

全过  → 200  {"status":"healthy","checks":{...全 true}}
任一挂 → 503  {"status":"degraded","checks":{...指明哪项 false}}

注意：
- model_warm 可放进 body 当**可选 info 字段**（反正 eager load 下几乎恒真，信息价值低），不 gate 200/503。
- "池 worker 活着" 不单列：池真死由 errors_total{kind=engine_error} 升 + inflight 卡住体现。
- ⚠️ **冷启动盲区（codex #5/#6，记 TODO）**：socket 在模型加载后才绑，若 `initialize()` 卡死/失败则 /health 根本答不了。本 PR **不解决**（启动卡死罕见 + 日志/PM2 兜底）；要冷启动可观测得动 main.py 启动序，记 TODO 独立做。
```

客户端打转录前可先 `GET /health` 预检；运维 curl 一眼看死活 + 是哪个子系统挂。

### 4.4 端点响应约定

| 端点 | 方法 | 格式 | 鉴权 | 内容 |
|---|---|---|---|---|
| `/health` | GET | JSON | 无（聚合状态，无敏感信息） | 状态机结果，200/503 |
| `/metrics` | GET | Prometheus text | 可选 token（D6） | §4.2 全指标 |
| `/` | GET | HTML | 无 | **已实现（2026-06-17）**：极简状态页，静态 HTML/JS，fetch /health + /metrics 渲染人读表格 + 每 3s 自刷。token 由用户在 URL `?token=` 提供（不嵌服务端 HTML，A5）。浏览器打开 `http://host:端口/` 即仪表盘 |

### 4.5 安全边界（D6）

- /metrics **只出聚合计数**，代码评审钉死：禁止任何 file_hash / 文件名 / 音频内容 / 用户标识进指标
- **A5 + codex #16 已核实**：`server.host` 默认就是 `0.0.0.0`（不是 LAN 限定）。所以"可选 token + LAN 可信"是**坏默认**——/metrics 会在所有绑定网卡上裸暴露。
- 修正：**`server.host=0.0.0.0` 且 `metrics_token` 未设时，/metrics 拒绝服务并启动期 warn**（强制要么设 token，要么显式绑 LAN/loopback）。/metrics 校验 `Authorization` header 或 query token（注意 #2 query 解析）。
- /health 裸放（只是死活，无敏感信息）。钉测试：/metrics 响应**不回显 `metrics_token` / 任何 config secret**；0.0.0.0+无 token 时 /metrics 返 403/404。
