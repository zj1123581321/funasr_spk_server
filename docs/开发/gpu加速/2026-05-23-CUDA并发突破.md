# Qwen3 ort_cuda + pool=2 并发突破 — 单进程多实例方案

**Date**: 2026-05-23
**Branch**: `spike/qwen3-diarize-poc`
**Sprint**: Qwen3 dev box (RTX 3060) 上 ort_cuda backend + pool_size=2 并发
**Dev box**: `vm200-i7-3060llm` (Tailscale `100.103.92.95`, RTX 3060 12G, Ubuntu 24.04, 8 vCPU)

## TL;DR

**问题**: 上一个 sprint 落档 ([2026-05-22-ORT-CUDA-diarize-backend.md](./2026-05-22-ORT-CUDA-diarize-backend.md)) 验证 2 进程并发 ort_cuda backend 撞 CUDNN cross-process race (TitaNet LSTM 在 cudnnRNNForward illegal memory access), 单进程 RTF 0.047 但 pool_size 受限于 1.

**方案**: 放弃多进程, 改单进程内 N 个独立 `Qwen3DiarizeTranscriber` 实例 + `asyncio.gather` 并发. 同进程内多个 `llama_context` + 多个 ORT `InferenceSession` **共享同一 cuda context, race-free** (llama.cpp + ORT 都支持这种 pattern).

**实测 (RTX 3060 + 8 vCPU)**:

| audio | 单 task wall | 单 task RTF | 并发 wall | 并发 RTF | 是否达标 |
|---|---|---|---|---|---|
| 60s | 3.65s | 0.061 | **5.69s** | **0.095** | RTF ✅ (spk over-merge known issue) |
| 300s | 15.01s | 0.050 | **25.28s** | **0.084** | ✅ |
| **1800s** | **85.20s** | **0.047** | **141.95s** | **0.079** | **🎯 全过** |

## 探索过程: 5 个候选方案的实测

### ❌ Candidate 1a: CUDA MPS daemon (default thread% = 100)

- 启 `nvidia-cuda-mps-control -d`, daemon 在第一个 CUDA app 接入时正确 fork 出 MPS server
- 复跑 `_remote_concurrent_probe.sh` (2 个独立 python 进程, 各自起 transcriber)
- **结果**:
  - Worker 1: wall=108.11s, RTF=0.060 (慢于单任务 baseline 0.047, 退步 27%)
  - Worker 2: 撞死在 LLM decode 阶段 — `GGML_ASSERT(i01 >= 0 && i01 < ne01) failed` (ggml_compute_forward_get_rows, CPU 端 token id 越界)
  - 错误**变了形式但仍撞** — 不再是上次的 cudnnRNNForward illegal memory access, 改成 LLM token id 越界 (推断: MPS 让 2 worker 共享 cuda context, ASR encoder 输出在 GPU buffer 间被改, 解码出非法 token id)

### ❌ Candidate 1b: CUDA MPS daemon (active thread% = 50)

- `echo set_default_active_thread_percentage 50 | nvidia-cuda-mps-control`
- 每 worker 限制 50% SM 使用率
- **结果**:
  - Worker 1: wall=120.89s, RTF=0.067, speakers=['Speaker1', 'Speaker3'] **+ diarize ID 错乱**
  - Worker 2: 同样 `ggml_get_rows i01<ne01 failed`
  - Total wall: 121.35s
- **结论**: MPS thread% 调小不解 race. 跨进程 cuda context 共享 buffer 是 nvidia driver/cuDNN 跨进程 isolation 本质问题, MPS 不解.

### ✅ Candidate 3: 单进程 + N transcriber 实例 + asyncio.gather

**核心洞察**:

| 多进程 (撞) | 单进程多实例 (work) |
|---|---|
| 2 个 OS process, 各自 cuda context | 1 个 OS process, 1 cuda context |
| cuDNN handle 跨进程共享 (race) | cuDNN handle 单 context 内有序 |
| LLM ctypes 加载 libllama.so 各进程独立 (但 GPU buffer 跨进程) | 同进程多 `llama_context` 各自 alloc, GPU stream 单 context 序列化 |
| ORT InferenceSession 跨进程 GPU mem race | 同进程多 InferenceSession thread-safe |

**实现**: `scripts/_remote_single_proc_concurrent_probe.py` — 直接 `new` 2 个 `Qwen3DiarizeTranscriber`, 各自 `initialize()` 后用 `asyncio.gather([tx1.transcribe(audio), tx2.transcribe(audio)])` 跑.

**为什么 race-free**:
1. **llama.cpp 设计支持单进程多 context**: 各 `llama_context` 独立 alloc CUDA buffer + KV cache, GPU kernel 提交到同一 cuda context 的 default stream, 自然序列化执行
2. **ORT CUDA EP 多 session thread-safe**: 同进程内多个 `InferenceSession` 调用并发 OK, CUDA kernel 走 EP 内部 stream
3. **Python ctypes + GIL**: 各 ASR/diarize 调用通过 `loop.run_in_executor(None, ...)` 推到 default ThreadPool (12 workers on 8 vCPU box), 真正多线程并发, 但 ctypes 调 C 函数时 GIL release, OK

### 300s + 60s 数据

| audio | TOTAL_WALL | task-1 RTF | task-2 RTF | speakers | scaling vs single |
|---|---|---|---|---|---|
| 60s | 5.69s | 0.095 | 0.094 | 1 (over-merge, known) | 1.56x |
| 300s | 25.28s | 0.084 | 0.084 | 2 ✅ | 1.68x |
| 1800s | 141.95s | 0.079 | 0.079 | 2 ✅ | 1.67x |

**Scaling 解读**: 2 task wall / 1 task wall ≈ 1.67x (1800s/300s 稳定). 完美并行是 1.0x, 完全串行是 2.0x. 实际 1.67x 意味着:
- LLM decode 阶段: GPU SM 共享, 2 context 的 kernel 实际 ~70% 并行 (单进程内 stream 序列化但 SM 复用)
- ASR encoder + diarize: 同进程内 ORT thread pool 真并行, 接近 1.0x 加速
- 整体 throughput = 2 / 1.67 = 1.20x 单任务 throughput

## 工程意义

- **RTX 3060 12G + 8 vCPU 上 ort_cuda pool=2 可行**: 单进程内 2 个 transcriber 实例 + asyncio.gather, total wall 141.95s vs 串行 2x 单任务 170.4s, **省 17%**
- **零侵入 production**: Mac 路径 (sherpa backend) 完全未动, Linux + CUDA dev box 上跑新 probe script 验证
- **vs MPS 路径**: MPS 试了两种 thread% 设置 (100% / 50%) 都撞 (Worker 2 在 LLM decode token id 越界), 单进程多实例方案绕开了跨进程 cuda context 共享问题
- **Throughput 加速**: 2 task 并发 throughput = 1.20x 单 task throughput; 但单 task RTF 0.079 vs 单跑 0.047, 退化 68% (LLM 占 wall 90s 是主导, 同时 2 context decode SM 抢占减速)

## 已知 caveat

1. **60s 短音频 over-merge 到 1 speaker** — 跟 ort_cuda backend 单 diarize 路径同一个问题 (`docs/开发/gpu加速/2026-05-22-ORT-CUDA-diarize-backend.md` 已知遗留), 跟 concurrent 方案无关. 长音频 (300s+) 不触发.
2. **本 sprint 仅落档 probe script, 未把 in-proc gather 接到 production `qwen3_pool_transcriber.py`** — 当前 pool_transcriber 仍是 file-based multiprocess queue, runtime=cuda 时 pool_size=2 仍走旧路径 (会撞). 接 in-proc 路径需要新 dispatcher (Mac runtime → 旧 multiprocess pool; CUDA runtime → in-proc gather), 估工 ~150 行新代码 + parity 测试, 留下个 sprint.
3. **Mac unit test pre-existing 7 fail** — 跟 Qwen3 ASR encoder `COREML_ANE_FULL` 默认值有关, 跟本 sprint 改动无关 (本 sprint 仅加 scripts/, 未动 src/). 不阻塞本 sprint 交付.
