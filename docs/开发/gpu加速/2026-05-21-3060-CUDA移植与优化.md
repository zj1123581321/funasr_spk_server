# 3060 + CUDA 移植与性能优化

**Date**: 2026-05-21
**Branch**: `spike/qwen3-diarize-poc`
**Author**: smoke test sprint on dev box `vm200-i7-3060llm` (Tailscale `100.103.92.95`)

## 目标

把当前 macOS-only 的 Qwen3-Diarize 引擎跑通在 Linux + RTX 3060 12G 上, 实测 RTF 与 M1 Max baseline 对比.

## 环境

| 项 | 值 |
|---|---|
| GPU | RTX 3060 12GB (Driver 570 / CUDA Runtime 12.8) |
| CPU | i7-12700KF, **KVM 限 4 vCPU** (物理是 12c20t) |
| OS | Ubuntu 24.04 (kernel 6.8) |
| Python | 3.12.3 |
| 系统 CUDA toolkit | `nvcc 12.0` + `nvidia-cuda-dev:amd64` |

## 关键结果(60s + 5min 同 fixture, JSON mode)

| 配置 | 60s wall | 60s wall RTF | 5min wall | 5min wall RTF |
|---|---|---|---|---|
| **CPU-only baseline (sherpa CPU, 8 threads)** | 13.5s | 0.225 | - | - |
| CUDA encoder (LD_LIBRARY_PATH 修复, sherpa CPU 8t) | 11.5s | 0.183 | - | - |
| **sherpa num_threads=2** (sweet spot) | **7.18s** | **0.030 (ASR)** | 33.16s | 0.111 |
| **sherpa num_threads=4** (长音频最优) | 6.99s | 0.090 (ASR) | **28.09s** | **0.094** ✨ |
| sherpa num_threads=8 (默认) | 13.3s | 0.191 | 58.28s | 0.194 ❌ |
| **M1 Max baseline (参考)** | - | - | - | **0.118** |

**结论:RTX 3060 12G(4 vCPU)上 wall RTF 0.094 比 M1 Max baseline 0.118 快 20%**, 而且工程上无需自编 sherpa-onnx CUDA build.

## 工程改造点(对当前 main 分支的 delta)

### 1. `src/core/qwen3/asr.py` — provider mapping 加 cuda/tensorrt 分支

原 mapping 只支持 `cpu/coreml_*/auto`, Linux 上 `FUNASR_QWEN3_ASR_ENCODER_PROVIDER=cuda` 落入 `else` → `CPU`. 已修改:

```python
if cfg_value == "cpu":         onnx_provider = "CPU"
elif cfg_value == "cuda":      onnx_provider = "CUDA"          # NEW
elif cfg_value in ("tensorrt", "trt"):  onnx_provider = "TENSORRT"  # NEW
elif cfg_value == "coreml_ane_fe":      onnx_provider = "COREML_ANE_FE"
elif cfg_value == "coreml_ane_full":    onnx_provider = "COREML_ANE_FULL"
else:  # auto / 未知值
    onnx_provider = "COREML_ANE_FE" if _sys.platform == "darwin" else "CPU"
```

### 2. `src/core/vendor/qwen_asr_gguf/inference/encoder.py` — warmup random data

非 DML 模式 warmup 原本用 `np.zeros(2.0s)` , 修改为 `np.random.standard_normal(dml_pad_to s) * 0.1`. 改动**没改善 mel cold start**, 真正根因是 sherpa thread contention(见后文), 但 warmup 用真实尺寸 + non-zero data 仍是 best-practice.

同时加了一个**可选 timing log**:

```bash
FUNASR_QWEN3_ENCODER_TIMING=1 python -m pytest tests/integration/test_qwen3_diarize_e2e.py
# 输出: [encoder-timing] mel=Xms frontend=Yms backend=Zms total=Tms
```

### 3. Linux GPU 依赖装载(不入仓, 文档化在 .env / 部署 doc)

#### 3.1 llama.cpp CUDA build

`vendor/qwen_asr_gguf/inference/bin/` 原本只有 macOS dylib. Linux 需要 self-compile llama.cpp + GGML CUDA:

```bash
git clone --depth 1 https://github.com/ggml-org/llama.cpp.git ~/llama.cpp
cd ~/llama.cpp
cmake -B build -DGGML_CUDA=ON -DLLAMA_BUILD_TESTS=OFF \
      -DLLAMA_BUILD_EXAMPLES=OFF -DLLAMA_BUILD_TOOLS=OFF \
      -DBUILD_SHARED_LIBS=ON -DCMAKE_CUDA_ARCHITECTURES=86  # 86 = Ampere (3060)
cmake --build build --config Release -j$(nproc)

DEST=<project>/src/core/vendor/qwen_asr_gguf/inference/bin
find build \( -name "libllama.so" -o -name "libggml*.so" \) -exec cp -v {} "$DEST/" \;
# 关键 .so: libllama.so libggml.so libggml-base.so libggml-cpu.so libggml-cuda.so
```

vendor `inference/llama.py` 已经有 `if sys.platform == "darwin"... else .so` 分支, 把 `.so` 放进 bin/ 即可被 `_preload_dlls` 加载.

#### 3.2 Python 依赖

`requirements.txt` 里 `onnxruntime>=1.14.0` → 装 `onnxruntime-gpu>=1.19.0`:

```bash
pip install -r requirements.txt
pip uninstall -y onnxruntime
pip install "onnxruntime-gpu>=1.19.0"
```

注意要装 `python3.12-venv`(Ubuntu 默认不带).

#### 3.3 `LD_LIBRARY_PATH` 必须显式 export

**这是隐形大坑**: onnxruntime-gpu 1.26 要求 `libcudnn.so.9`(在 venv 内的 `nvidia/cudnn/lib/`), 但 `LD_LIBRARY_PATH` 默认不指向 venv. 不设的话 ORT silent fallback CPU(stderr 报 `Failed to load libcudnn.so.9`, 但 Python 进程不抛异常).

部署/启动 script 必须设:

```bash
NV=venv/lib/python3.12/site-packages/nvidia
TRT=venv/lib/python3.12/site-packages/tensorrt_libs   # 装了 tensorrt-cu12 才有
export LD_LIBRARY_PATH=$TRT:$NV/cudnn/lib:$NV/cublas/lib:$NV/cufft/lib:\
$NV/cuda_runtime/lib:$NV/cuda_nvrtc/lib:$NV/nvjitlink/lib:\
$NV/cusparse/lib:$NV/curand/lib:$NV/cusolver/lib:$NV/cuda_cupti/lib:$LD_LIBRARY_PATH
```

参考 `scripts/_remote_run_provider.sh` 作为辅助脚本.

## 性能调优(主要 win)

### 优化 1: `FUNASR_QWEN3_NUM_THREADS` 必须 ≤ vCPU/2

**根因**: ASR + diarize 通过 `asyncio.gather` 真并行. sherpa-onnx CPU build 跑 pyannote-seg + TitaNet 占满 num_threads 个 BLAS/OMP thread. 在 4 vCPU 下:

| sherpa num_threads | 60s chunk 1 mel | 60s wall | 5min wall |
|---|---|---|---|
| 1 | 19 ms | 9.19 s | - |
| **2** | **19 ms** | **7.18 s** | 33.16 s |
| **4** | 3572 ms | 6.99 s | **28.09 s** ✨ |
| 8 (默认) | 9689 ms | 13.3 s | 58.28 s ❌ |

`num_threads=8` 在 4 vCPU 上 oversubscribe, ASR 的 numpy.fft 第一次跑 40s 真实音频时, thread pool 被 sherpa 完全挤占, mel 计算等了 9.5s 才轮到. 这就是之前 RTF 0.225 的 90% 时间.

**推荐**:
- 短音频(<1min)/多并发 → `FUNASR_QWEN3_NUM_THREADS=2`
- 长音频(>3min)/单任务 → `FUNASR_QWEN3_NUM_THREADS=4`(只有 chunk 1 mel 慢一次, 长音频里被摊薄)

config.py default 仍是 `num_threads=4`(Mac 10 cores 上 PoC 验证最优), Linux 4 vCPU 上需手动 env override.

### 优化 2: ASR encoder provider 选 CUDA(微弱收益)

`FUNASR_QWEN3_ASR_ENCODER_PROVIDER=cuda` 让 frontend + backend ONNX 走 CUDA EP. 实测 encoder inference ~130ms/chunk × 2 = 260ms(60s 音频). 比 CPU encoder 1.2s × 2 = 2.4s 略快, 但 ASR 总耗时(含 mel + chunk + LLM)主要被 LLM token gen 主导, encoder 优化在长音频上摊薄不明显.

`FUNASR_QWEN3_PROVIDER` 是 sherpa diarize provider, 保持 `cpu`(GPU 版有 segfault, 见 known issues).

## Known Issues / 不可行路径

### TRT EP 不支持 Qwen3-ASR encoder

`backend.onnx`(158 MB)用了 ORT contrib op `MatMulNBits`(4-bit 量化 MatMul)+ `BiasGelu`. TensorRT 10.9 不认这些 op, ORT TRT EP 解析模型时报:

```
ERROR: IPluginRegistry::getCreator: Cannot find plugin: MatMulNBits, version: 1
ERROR: In node ... INVALID_NODE: creator && "Plugin not found"
```

要走 TRT 必须重新导出一份纯标准 ONNX op 的 encoder(FP16/FP32 MatMul 替换 MatMulNBits), 工作量大且会丢 4-bit 压缩, 收益不明.

### sherpa-onnx CUDA build 跟 llama.cpp CUDA segfault

prebuilt GPU wheel 可拿到(`pip install https://huggingface.co/csukuangfj2/sherpa-onnx-wheels/resolve/main/cuda/1.13.2/sherpa_onnx-1.13.2+cuda12.cudnn9-cp312-cp312-linux_x86_64.whl`), sherpa 单跑 OK(60s diarize 3.35s on GPU). 但跟 llama.cpp CUDA 同 process 共存时 SIGSEGV(core dump), 在 LLM warmup 完成 → ASR/diarize 启动之间崩.

推测是 CUDA context 跨 lib 不兼容(llama.cpp libggml-cuda.so 跟 onnxruntime-cuda.so 在同一 cuda context 上 set state 互踩). 没深挖.

**Workaround**: 装 sherpa GPU wheel 但配置 `FUNASR_QWEN3_PROVIDER=cpu`, sherpa-onnx 仍走 CPU path, 跟 LLM CUDA 共存稳定. RTF 不受影响.

## ORT CUDA 直 wrap 验证(2026-05-21 续, 候选 1 PoC 通过)

针对上面的 sherpa-onnx CUDA segfault, 验证假设: **冲突在 sherpa-onnx 的 C++ wrapper 层, 不在 ONNX Runtime CUDA EP 本身**. 改用 Python `onnxruntime.InferenceSession` 直接跑同一份 pyannote-seg + TitaNet ONNX 模型, 跟 llama.cpp CUDA 同 process 共存.

### 验证脚本

`scripts/_remote_ort_cuda_clash_check.py` — 跑三种顺序:
- `--order=llm_first`: LLM init → ASR full pipeline → ORT CUDA pyannote-seg/titanet init + inference
- `--order=ort_first`: ORT 先 → LLM 后
- `--order=interleaved`: 所有 init 先做完再交错 inference

### 结果

| 顺序 | LLM CUDA init | LLM 完整 ASR inference | ORT pyannote-seg CUDA init+inference | TitaNet CUDA init | segfault? |
|---|---|---|---|---|---|
| llm_first | ✅ | ✅ 3.96s, RTF 0.066, 真 text 368 字符 | ✅ 178.8 ms | ✅ | ❌ 无 |
| interleaved | ✅ | ✅ 4.27s, RTF 0.071 | ✅ 173.3 ms | ✅ | ❌ 无 |

(TitaNet 真实 inference 没跑成, 是脚本喂错输入 — TitaNet ONNX 期望 `(B, 80, T_mel)` log-mel features, 不是 raw audio, 这是工程细节不影响共存结论.)

**结论**: ORT CUDA EP **跟 llama.cpp CUDA 完全共存稳定**. sherpa-onnx 的 C++ wrapper 层是冲突源, 不是 ORT 本身.

### 性能预估

- pyannote-seg CUDA inference: **178 ms / 10s chunk** → 60s 音频 6 chunks ≈ **1s** (vs sherpa CPU 13s, 13x 加速)
- TitaNet CUDA 同量级, < 1s
- **diarize 总耗时**: 60s 音频 < 2s (vs sherpa CPU ~15s), 30min 音频 < 10s (vs CPU 100s)
- **wall RTF 预期**: 当前 0.090 → **~0.04-0.05** (sherpa 不再是瓶颈)

### 工程实现路径

把 sherpa-onnx 的 `OfflineSpeakerDiarization` 拆成 4 个独立组件用 Python 重写:

1. **pyannote-seg ORT CUDA inference**(已验证可跑): 输入 10s audio chunk → 输出 powerset speaker activity (1, 589, 7), 滑动窗 + Whisper-style 平均得到完整 audio 的 speaker activity 时序
2. **TitaNet 80-band log-mel preprocessing**: numpy/torchaudio 实现, sample_rate=16k, win=400, hop=160, n_mels=80
3. **TitaNet ORT CUDA inference**: 吃 mel (B, 80, T), 出 192-dim speaker embedding
4. **FastClustering Python 实现**: 凝聚聚类 + cosine 距离 + threshold cut, sherpa 算法开源可参考 `k2-fsa/sherpa-onnx/sherpa-onnx/csrc/fast-clustering.cc`

集成思路: 跟现有 `sherpa-onnx` backend 并存, 通过 runtime detect 切换:

```python
# 设计层面
class DiarizeBackend(Protocol):
    async def run(self, audio_path) -> List[Turn]: ...

class SherpaDiarizeBackend(DiarizeBackend): ...      # Mac / Linux CPU fallback (现状)
class OrtCudaDiarizeBackend(DiarizeBackend): ...     # Linux CUDA (新增)

# runtime detect (新增 src/core/runtime.py):
if sys.platform == "darwin": backend = "sherpa"
elif _cuda_available_and_libs_ok(): backend = "ort_cuda"
else: backend = "sherpa"
```

实现工作量约 1-1.5 天, 验收标准: 跟 sherpa CPU baseline 跑同一份 fixture, speaker 数 / segment 时间戳差异 < 5%.

## 30min 超长音频 RTF

| 配置 | wall | wall RTF | ASR encode 耗时 |
|---|---|---|---|
| num_threads=2 | 198.42 s | 0.110 | 5.78 s |
| **num_threads=4** | **161.26 s** | **0.090** ✨ | 5.84 s |

ASR encoder 在 30 min 音频上稳定 5.8s(45 个 chunk × ~130ms), wall 大头是 sherpa diarize CPU(threads=2 ~138s, threads=4 ~103s). 30 min RTF 0.090 跟 5min 0.094 几乎一致, 说明 init overhead 在 5 min 以上音频已经完全摊薄, RTF 渐近值就是 0.09.
