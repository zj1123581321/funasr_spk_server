# Qwen3 词级时间戳 — PoC 计划

> 由 `/plan-ceo-review`(2026-06-09)产出。前置交接文档:`docs/开发/2026-06-09-qwen3-词级时间戳-设计讨论-新session-prompt.md`。
> 目标:用最小代价把 A/B/C 三条路线的关键未知数变成数据,**选型靠数据不靠拍脑袋**。

## 背景一句话
ASR 已输出整段全文,缺的是词级时间戳。补它 = 再跑一个 forced aligner 吃 `(audio, 全文)` → 词级 items。候选:重的官配 Qwen aligner(精度最高) vs 轻的外挂 FunASR fa-zh(CPU 可跑、长音频安全)。

## CEO 评审已确立的事实
- **B 路线权重可得**:三件套在用户的 `CapsWriter-Offline-with-AI`(deploy/linux-vulkan),~813MB(frontend.int4.onnx 22.8MB + backend.int4.onnx 239MB + **llm.q5_k.gguf** 551MB)。
- **接入坑**:① 本 vendor `AlignerConfig` 默认 `llm_fn=q4_k`,实际是 **q5_k** → 必须 override;② 本 vendor `n_ctx=2048`(别处 3072)→ 单次 align ≈68s 上限,窗口要压到 **~55-60s**;③ 长音频(60/81min)A、B 都**必须自建 VAD 切块 + 跨块拼接**,`align()` 已接受 `offset_sec`。
- 详见记忆 `reference_qwen3_aligner_weights`。

---

## 共用基础设施(先搭)
- **eval 音频**:沿用现有 diarize eval 集,挑 ① 1 个短中文 clip(<60s) ② 1 个 60min 长中文 clip。
- **指标口径**(deep research 共识):**AAS**(Average Absolute Shift,平均绝对偏移 ms,越小越好)、边界命中率(t≤50ms / ≤100ms)、**RTF**(Mac MPS 实测)。
- 落地目录:`spikes/qwen3_word_timestamp/`(照 spikes/ 惯例,出 SUMMARY.md)。

---

## PoC-A — Qwen vendor aligner(重/高精度路线)
1. 拷三件套到 `models/qwen3_diarize/Qwen3-ASR-1.7B/`(或 sibling aligner 目录)。
2. 走 `build_engine_config(enable_aligner=True)`,override `align_config.llm_fn → qwen3_aligner_llm.q5_k.gguf`,encoder onnx_provider 先 CPU。
3. 短 clip 跑通 → 确认 `WordItem` 非空、目测时间戳对、量单次 RTF。
4. **建切块层**:VAD 或固定 ~55s 窗 + overlap,逐窗 `align(audio_chunk, text_chunk, offset_sec)`,按 offset 拼接 + 去重。
5. 60min 长 clip → 确认不崩(护住 n_ctx)、跨块边界对齐不漂。
6. 量 Mac MPS 全程 RTF(encoder CPU/ANE + GGUF Metal)→ diarize 开关经济账用此数。
- **判定**:短 clip AAS、长 clip 不崩 + 跨块 AAS、RTF 增量。

## PoC-B — FunASR fa-zh(轻量路线)
1. 独立接 fa-zh(38MB)+ fsmn-vad,同 clip 喂 `(audio, ASR全文)` → 字符/词级戳。
2. 量 RTF(报告称 CPU RTF 0.0076,实测确认)、目测中文准度。
3. 60min 长 clip 确认不崩(fsmn-vad 内置,天然安全)。
- **判定**:中文 AAS、RTF、长音频 OK。
- 备选:多语场景留 MMS-300M CTC-FA,本轮不做。

## PoC-C — 现状近似误差量化(baseline)
- 用 A/B 中更准的当 gold,量现有 `silence_align + 字符比例` 切句的 AAS 偏移。
- **产出**:词级精确到底比现状好多少 → 决定值不值得引入 aligner。

## 头对头 + 选型
出一张表:`{aligner, AAS_zh, 边界命中率, RTF_mac, 长音频OK, 依赖栈}`,A vs B vs 现状(C)。据此选 forced-align backend。

---

## PoC 之后(落地方向,本计划不含)
- 选定 backend → 照 diarize 的 rule-of-two runtime-aware dispatch 模式新建 forced-align 模块,词级 items 旁路掉 `merge_asr_chunks_and_diarize` 的字符比例近似。
- diarize 开关议题塌缩成纯布尔(时间戳恒有基线)。
- 严格 TDD(红→绿→commit),改 Qwen3 路径后跑 parity(`FUNASR_RUN_INTEGRATION=1`)。

## 风险/未决
- A 启用 = 进程内第二个 llama.cpp GGUF(跟 ASR GGUF 并存),in-proc pool 内存/CUDA 行为待 PoC 验证。
- fa-zh 中文准度未实测,可能不及 Qwen aligner 的 AAS 33ms。
- C 若证明现状够用,A/B 都可不落地。
