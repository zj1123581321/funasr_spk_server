# FunASR WebSocket 客户端交互指南

## 概述

本文档详细介绍如何与 FunASR 转录服务器进行 WebSocket 通信，包括单文件上传、分片上传、实时转录状态监控等功能。服务器已根据最佳实践进行优化，支持大文件的稳定传输。

## 连接配置

### WebSocket 连接参数

```python
import asyncio
import websockets
import json
import base64
import hashlib
from pathlib import Path

# 推荐的连接配置（适配服务器端设置）
websocket = await websockets.connect(
    "ws://localhost:8767",
    ping_interval=60,       # 60秒发送一次心跳（与服务器一致）
    ping_timeout=120,       # 心跳响应超时120秒
    close_timeout=60,       # 关闭连接超时60秒
    max_size=10 * 1024 * 1024,  # 单消息最大10MB
    read_limit=2**20,       # 1MB读缓冲
    write_limit=2**20       # 1MB写缓冲
)
```

## 文件上传流程

### 1. 单文件上传（< 5MB）

适用于小文件的快速上传。

#### 步骤1：发送上传请求

```json
{
  "type": "upload_request",
  "data": {
    "file_name": "audio.mp3",
    "file_size": 1048576,
    "file_hash": "d41d8cd98f00b204e9800998ecf8427e",
    "output_format": "json",  // 可选: "json" 或 "srt"
    "force_refresh": false,   // 可选: 是否强制刷新缓存
    "engine": "funasr",       // 可选(PR1新增): 选择 ASR 引擎
    "language": "chi",        // 可选: 识别语言 ISO 码 (chi/eng/jpn/kor…), 驱动词级时间戳语言
    "diarize": true           // 可选: 是否输出说话人区分 (默认 true, 完全向后兼容)
  }
}
```

> **engine 字段（PR1 新增）**
> - 省略 → 走服务端 `FUNASR_DEFAULT_ENGINE`（默认 `funasr`），与 PR1 前完全兼容
> - `"funasr"` → 基于 FunASR + cam++ 说话人识别（生产稳定路径）
> - `"qwen3"` → Qwen3-ASR 1.7B + sherpa/ORT diarize（当前生产主路径）
> - 缓存 key 包含 engine，同一文件在不同引擎下各自缓存，不会互相覆盖

> **diarize 字段（diarize 开关）**
>
> `diarize=false` ⇒ 响应不含说话人区分。**不只是少个字段，输出形态与分段边界语义都会变化**：
>
> - **JSON**：每个 segment 的 `speaker` 为 `null`（表示"未做区分"，与"真只有一人 Speaker1"语义可区分），顶层 `speakers` 为 `[]`
> - **SRT**：字幕行**不带 `SpeakerN:` 前缀**（纯文本行）
> - **分段边界语义变化**：diarize=true 时分段按说话人 turn 切；diarize=false 时没有 turn 边界，分段来自 ASR 内部 chunk 时间窗，再经服务端分层切分（有词级时间戳按词隙切，否则按静音切）成字幕可用粒度。**两种模式的分段数量与切点位置不可比**
> - **性能**：qwen3 引擎真跳过说话人聚类与相关后处理，省算力；funasr 引擎照常计算、出口投影抹除
> - **缓存**：qwen3 的 diarize=false 结果独立缓存（key 折 `+nospk` 维）；若同文件已有带说话人的缓存，服务端会现场投影直接返回（响应 metadata 中 `projected: true`），不重算
> - **`num_speakers` 配置**：diarize=false 时服务端 `num_speakers` 配置闲置不生效（服务端日志会提示）
> - **部署顺序约束**：老版本 server 会忽略未知的 `diarize` 字段（静默照常带 speaker）——必须 **server 先升级、客户端后启用此字段**
>
> **结果 metadata 回显**：转录结果带 `metadata` 块回显本次请求实际生效的选项：
> `{"engine", "diarize", "word_align", "language", "projected"}`。
> 合并优先级：request 字段 > 分片 session 回填 > 服务端 config > 引擎默认。
> SRT 响应没有 JSON 结果载体，`metadata` 挂在 `result` payload 顶层。

#### 步骤2：接收上传就绪响应

```json
{
  "type": "upload_ready",
  "data": {
    "task_id": "uuid-task-id",
    "message": "准备接收文件数据"
  }
}
```

#### 步骤3：发送文件数据

```json
{
  "type": "upload_data",
  "data": {
    "task_id": "uuid-task-id",
    "file_data": "base64编码的文件数据"
  }
}
```

#### 步骤4：接收上传完成响应

```json
{
  "type": "upload_complete",
  "data": {
    "task_id": "uuid-task-id",
    "message": "文件上传成功，开始处理"
  }
}
```

### 2. 分片上传（≥ 5MB）

适用于大文件的稳定传输，自动分片处理。

#### 步骤1：发送分片上传请求

```json
{
  "type": "upload_request",
  "data": {
    "file_name": "large_audio.m4a",
    "file_size": 52428800,    // 50MB
    "file_hash": "文件MD5哈希",
    "chunk_size": 1048576,    // 1MB分片大小
    "total_chunks": 50,       // 总分片数
    "upload_mode": "chunked", // 标识分片上传模式
    "output_format": "json",
    "force_refresh": false,
    "engine": "funasr",       // 可选(PR1新增): 同单文件上传，分片场景同样支持
    "language": "chi",        // 可选: 同单文件上传
    "diarize": true           // 可选: 同单文件上传 (session 创建时记录, 完成时回填)
  }
}
```

> 分片上传场景下 `engine` / `language` / `diarize` 字段在 session 创建时记录，所有分片传完后用同一组选项处理。

#### 步骤2：接收分片上传就绪响应

```json
{
  "type": "upload_ready",
  "data": {
    "task_id": "uuid-task-id",
    "message": "准备接收分片数据",
    "chunk_size": 1048576,
    "total_chunks": 50
  }
}
```

#### 步骤3：循环发送分片数据

```json
{
  "type": "upload_chunk",
  "data": {
    "task_id": "uuid-task-id",
    "chunk_index": 0,         // 分片索引（从0开始）
    "chunk_size": 1048576,    // 当前分片实际大小
    "chunk_hash": "分片MD5哈希",
    "chunk_data": "base64编码的分片数据",
    "is_last": false          // 是否为最后一个分片
  }
}
```

#### 步骤4：接收分片确认响应

```json
{
  "type": "chunk_received",
  "data": {
    "task_id": "uuid-task-id",
    "chunk_index": 0,
    "progress": 2.0,          // 上传进度百分比
    "status": "received"      // 状态: "received" 或 "duplicate"
  }
}
```

#### 步骤5：接收最终上传完成响应

当所有分片上传完成后：

```json
{
  "type": "upload_complete",
  "data": {
    "task_id": "uuid-task-id",
    "message": "分片文件上传成功，开始处理"
  }
}
```

## 错误处理

### 错误响应格式

```json
{
  "type": "error",
  "data": {
    "error": "错误类型",
    "message": "详细错误信息"
  }
}
```

### 常见错误类型

| 错误类型 | 说明 | 处理建议 |
|---------|------|----------|
| `file_too_large` | 文件超过最大限制 | 检查文件大小限制 |
| `invalid_file_type` | 不支持的文件格式 | 使用支持的音频格式 |
| `hash_mismatch` | 文件哈希不匹配 | 重新计算并发送文件 |
| `chunk_hash_mismatch` | 分片哈希不匹配 | 重新发送该分片 |
| `session_not_found` | 上传会话不存在 | 重新开始上传流程 |
| `task_not_found` | 任务不存在（从未存在） | 凭 `file_hash` 重投（见异步轮询节） |
| `task_expired` | 任务曾存在但已被内存清理（超 TTL） | 凭 `file_hash` 重投，命中 DB 缓存秒回 |
| `queue_full` | 队列满（准入控制拒绝，非致命） | 据 `retry_after` 退避后重试（分片发 `finalize_upload` 不重传） |
| `too_many_sessions` | 并发上传会话达上限 | 稍后重试 |
| `missing_task_ids` | `task_status_batch` 未带非空 `task_ids` | 检查请求体 |

> ⚠️ `queue_full` 是带结构化字段的独立消息类型（`type:"queue_full"`，含 `retry_after`/`queue_size`/`max_queue_size`，并兼容 `error` 字段），不是致命错误，**务必识别并退避重试**，不要当成「服务挂了」。

## 批量任务与异步轮询契约（推荐姿势，TODOS #20）

> **2026-06-16 服务端落地。批量提交（10+ 任务）必须用本节姿势；单任务长音频沿用下一节「长音频」即可。**

### 为什么需要它

服务端真实并发 = `qwen3_pool_size`（默认 **1**，单 GPU）。批量提交 N 个任务时，队列里第 k 个要等约 `k × T`（T = 单任务处理时长）才轮到。客户端若在一条连接上 `recv()` **死等** `task_complete`，300s 接收窗只容得下前几个，**深队列后段注定超时**，误判「服务挂了」。

异步轮询**不提吞吐**（总耗时仍 `~N×T`），它解决的是「客户端不再因排队误判超时而挂掉」，且断连重连后仍能凭 `task_id` 拿结果。

### 核心原则：收 ack 后转轮询，不堵 recv

**上传协议零改动**——服务端入队后**照旧**回 `task_queued`（排队）或 `upload_complete`（直接开跑），都带 `task_id`。客户端的唯一改变是：

```
收到 task_queued / upload_complete  →  记下 task_id，不再堵等 task_complete，转入轮询循环
任何时刻收到 task_complete           →  直接终结该任务（缓存命中可能在 finalize 前就来，可当快路径）
```

### 批量轮询：必须用单条 task_status_batch

批量场景**禁止** per-task 各自轮询（N 条连接/N 条消息 = N+1 风暴）。用单条批量消息一次拿全集：

**请求**（`task_ids` 上限 **50**，超出服务端截断；>50 自行分批）：
```json
{ "type": "task_status_batch", "data": { "task_ids": ["id1", "id2", "id3"] } }
```

**响应**（逐 id 一项，完成项 `result` / `srt_content` **随响应内联**，不必再单查）：
```json
{
  "type": "task_status_batch",
  "data": {
    "items": [
      { "task_id": "id1", "status": "processing", "progress": 42.0, "result": null, "srt_content": null, "error": null },
      { "task_id": "id2", "status": "completed", "progress": 100.0, "result": { /* TranscriptionResult，JSON 格式 */ } },
      { "task_id": "id3", "status": "completed", "srt_content": "1\n00:00:00,000 --> ...", "result": null },
      { "task_id": "id4", "status": "failed", "error": "..." },
      { "task_id": "id5", "status": null, "error": "task_expired" },
      { "task_id": "id6", "status": null, "error": "task_not_found" }
    ]
  }
}
```

逐 id 判读规则：

| `status` | 含义 | 客户端动作 |
|---|---|---|
| `pending` / `processing` | 排队中 / 处理中（小帧，result=null） | 留在轮询集，下轮再查 |
| `completed`（JSON） | 完成 | 取 `result`，移出轮询集 |
| `completed`（SRT） | 完成 | 取 **`srt_content`**（不是 `result`！），移出轮询集 |
| `failed` / `timed_out` / `cancelled` | **终态** | 取 `error`，**停止轮询**该任务 |
| `null` + `error:"task_expired"` | 曾存在已被清理 | 凭 `file_hash` 重投（命中缓存秒回） |
| `null` + `error:"task_not_found"` | 从未存在 / 服务重启丢失 | 凭 `file_hash` 重投（分片需重传文件） |

> **务必识别终态全集**（`completed/failed/timed_out/cancelled`），只认 `completed` 会让失败任务被无限轮询。

### 参考实现（批量轮询循环）

```python
async def poll_batch(ws, pending_ids, *, first_delay, interval=8, max_rounds=10000):
    """对一批 task_id 轮询直到全部终结。

    pending_ids: 上传阶段从 task_queued/upload_complete 收集到的 task_id 集合
    first_delay: 首轮延迟，用入队 ack 给的 estimated_wait_seconds 估
    返回 {task_id: result_or_srt_or_error}
    """
    results, pending = {}, set(pending_ids)
    await asyncio.sleep(first_delay)
    for _ in range(max_rounds):
        if not pending:
            break
        # >50 自行分批；这里假设 ≤50
        await ws.send(json.dumps({
            "type": "task_status_batch",
            "data": {"task_ids": list(pending)[:50]},
        }))
        resp = json.loads(await ws.recv())
        if resp["type"] != "task_status_batch":
            continue  # 期间可能夹杂 task_complete 等 push，按需处理
        for it in resp["data"]["items"]:
            tid, st = it["task_id"], it["status"]
            if st == "completed":
                results[tid] = it.get("result") or it.get("srt_content")
                pending.discard(tid)
            elif st in ("failed", "timed_out", "cancelled"):
                results[tid] = {"error": it.get("error")}
                pending.discard(tid)
            elif st is None:  # task_expired / task_not_found
                results[tid] = {"poll_miss": it.get("error")}  # 交给上层凭 file_hash 重投
                pending.discard(tid)
            # pending/processing → 留在 pending，下轮再查
        if pending:
            await asyncio.sleep(interval)  # 退避，建议 5~10s
    return results
```

### poll-miss 兜底（重投）

轮询拿到 `task_expired` 或 `task_not_found` 时，凭 `file_hash` 重新走上传流程：

- **已完成的**：命中 DB 缓存秒回（同参数重发 `upload_request` 即收 `task_complete`，见「长音频」节的 `poll_result`）。
- **未完成 / 服务重启后**：分片的已落地文件在内存 session，**重启即失效，必须重传**（止血的「不重传」仅在 server 未重启的 `queue_full` 重试场景成立）。

### 与「长音频单任务」节的关系

- **单个长音频** → 沿用下一节「重发 `upload_request` 探测缓存」即可，简单够用。
- **批量 / 想要进度可见 / 想断连重连续取** → 用本节 `task_status_batch`。

完整协议规格见 `docs/开发/Server-Client 交互协议.md` 的「异步轮询契约」节。

## 长音频任务与连接断开（重要）

### 现象

服务端的说话人识别与部分后处理是同步 CPU 密集任务，长音频（约 20 分钟以上）处理期间可能阻塞 WebSocket 心跳响应，导致连接被掐断（客户端表现为 `ConnectionClosed` 等异常）。

### 关键事实：断连不影响任务

文件上传完成（收到 `upload_complete`）后，转录由服务端独立 worker 执行，**与连接生命周期无关**——连接断了任务照常跑完，结果写入服务端缓存。客户端不需要（也不应该）为此重新上传文件。

### 推荐客户端姿势

**短音频**（几分钟内）：处理通常在 1 分钟内完成，直接在当前连接上等 `task_complete` 即可。

**长音频**：不要 `recv()` 死等 `task_complete`，改用"上传即走 + 轮询探测"：

1. 收到 `upload_complete` 后**主动关闭连接**
2. 按 `音频时长 × 0.05~0.15`（经验 RTF 区间，随硬件浮动）估算处理时间，等待后重新连接
3. 用**与原请求完全相同的参数**（同 `file_hash`、同 `engine`、同 `diarize` / `language`、`force_refresh=false`）重新发送 `upload_request`：
   - 收到 `task_complete` → 任务已完成，这是缓存命中直接返回的完整结果
   - 收到 `upload_ready` → 任务尚未完成（缓存还没生成）。**此时不要发送文件数据**，直接关闭连接，等待下一轮重试
4. 轮询间隔建议 ≥ 30 秒，并设置重试上限

```python
async def poll_result(uri, file_hash, file_name, file_size,
                      interval=30, max_attempts=40, **req_options):
    """上传完成后轮询取结果: 同参数重发 upload_request, 命中缓存即返回.

    req_options 必须与原始上传请求一致 (engine/diarize/language/output_format),
    否则缓存 key 不同, 永远不会命中.
    """
    for _ in range(max_attempts):
        await asyncio.sleep(interval)
        async with websockets.connect(uri) as ws:
            await ws.send(json.dumps({
                "type": "upload_request",
                "data": {
                    "file_name": file_name,
                    "file_size": file_size,
                    "file_hash": file_hash,
                    "output_format": "json",
                    **req_options,
                },
            }))
            resp = json.loads(await ws.recv())
        if resp["type"] == "task_complete":
            return resp["data"]["result"]
        # upload_ready = 还没算完: 不发文件数据, 直接断开等下一轮
    raise TimeoutError("转录结果轮询超时")
```

### 两个注意事项

- **轮询时绝不要重新发送文件数据**：服务端对同 hash 的并发提交不做去重，重新上传会触发同一文件的重复计算，白白占用处理队列
- **轮询参数必须与原请求一致**：缓存按文件 hash + 引擎 + 选项（`diarize`、词级时间戳语言等）区分，参数不同视为不同结果，不会命中

## 完整客户端示例

### Python 客户端示例

```python
import asyncio
import websockets
import json
import base64
import hashlib
import os
from pathlib import Path

class FunASRClient:
    def __init__(self, server_url="ws://localhost:8767"):
        self.server_url = server_url
        self.websocket = None
    
    async def connect(self):
        """建立连接"""
        self.websocket = await websockets.connect(
            self.server_url,
            ping_interval=60,
            ping_timeout=120,
            max_size=10 * 1024 * 1024,
            read_limit=2**20,
            write_limit=2**20
        )
        
        # 接收欢迎消息
        welcome = await self.receive_message()
        if welcome.get("type") == "connected":
            print(f"✓ 连接成功: {welcome['data']['message']}")
            return True
        return False
    
    async def disconnect(self):
        """断开连接"""
        if self.websocket:
            await self.websocket.close()
    
    async def send_message(self, message):
        """发送消息"""
        await self.websocket.send(json.dumps(message, ensure_ascii=False))
    
    async def receive_message(self, timeout=30):
        """接收消息"""
        message_json = await asyncio.wait_for(
            self.websocket.recv(), timeout=timeout
        )
        return json.loads(message_json)
    
    def calculate_file_hash(self, file_path):
        """计算文件MD5哈希"""
        hash_md5 = hashlib.md5()
        with open(file_path, "rb") as f:
            for chunk in iter(lambda: f.read(8192), b""):
                hash_md5.update(chunk)
        return hash_md5.hexdigest()
    
    async def transcribe_file(self, file_path, output_format="json", force_refresh=False):
        """转录文件"""
        file_path = Path(file_path)
        file_size = file_path.stat().st_size
        file_hash = self.calculate_file_hash(file_path)
        
        print(f"文件: {file_path.name}")
        print(f"大小: {file_size/1024/1024:.2f}MB")
        print(f"哈希: {file_hash[:8]}...")
        
        # 判断使用单文件还是分片上传
        if file_size > 5 * 1024 * 1024:  # >5MB使用分片
            return await self._upload_chunked(file_path, file_size, file_hash, output_format, force_refresh)
        else:
            return await self._upload_single(file_path, file_size, file_hash, output_format, force_refresh)
    
    async def _upload_single(self, file_path, file_size, file_hash, output_format, force_refresh):
        """单文件上传"""
        print("使用单文件上传模式")
        
        # 发送上传请求
        request = {
            "type": "upload_request",
            "data": {
                "file_name": file_path.name,
                "file_size": file_size,
                "file_hash": file_hash,
                "output_format": output_format,
                "force_refresh": force_refresh
            }
        }
        
        await self.send_message(request)
        response = await self.receive_message()
        
        if response["type"] == "task_complete":
            # 直接返回缓存结果
            print("✓ 使用缓存结果")
            return response["data"]["result"]
        
        if response["type"] != "upload_ready":
            raise Exception(f"上传请求失败: {response}")
        
        task_id = response["data"]["task_id"]
        print(f"✓ 获得任务ID: {task_id}")
        
        # 读取文件并发送
        with open(file_path, 'rb') as f:
            file_data = f.read()
        
        upload_data = {
            "type": "upload_data",
            "data": {
                "task_id": task_id,
                "file_data": base64.b64encode(file_data).decode()
            }
        }
        
        await self.send_message(upload_data)
        response = await self.receive_message()
        
        if response["type"] != "upload_complete":
            raise Exception(f"文件上传失败: {response}")
        
        print("✓ 文件上传完成，等待转录结果...")
        
        # 等待转录完成
        return await self._wait_for_result()
    
    async def _upload_chunked(self, file_path, file_size, file_hash, output_format, force_refresh):
        """分片上传"""
        chunk_size = 1024 * 1024  # 1MB分片
        total_chunks = (file_size + chunk_size - 1) // chunk_size
        
        print(f"使用分片上传模式（{total_chunks}个分片）")
        
        # 发送分片上传请求
        request = {
            "type": "upload_request",
            "data": {
                "file_name": file_path.name,
                "file_size": file_size,
                "file_hash": file_hash,
                "chunk_size": chunk_size,
                "total_chunks": total_chunks,
                "upload_mode": "chunked",
                "output_format": output_format,
                "force_refresh": force_refresh
            }
        }
        
        await self.send_message(request)
        response = await self.receive_message()
        
        if response["type"] == "task_complete":
            print("✓ 使用缓存结果")
            return response["data"]["result"]
        
        if response["type"] != "upload_ready":
            raise Exception(f"分片上传请求失败: {response}")
        
        task_id = response["data"]["task_id"]
        print(f"✓ 获得任务ID: {task_id}")
        
        # 分片上传
        with open(file_path, 'rb') as f:
            for chunk_index in range(total_chunks):
                chunk_data = f.read(chunk_size)
                chunk_hash = hashlib.md5(chunk_data).hexdigest()
                
                chunk_message = {
                    "type": "upload_chunk",
                    "data": {
                        "task_id": task_id,
                        "chunk_index": chunk_index,
                        "chunk_size": len(chunk_data),
                        "chunk_hash": chunk_hash,
                        "chunk_data": base64.b64encode(chunk_data).decode(),
                        "is_last": chunk_index == total_chunks - 1
                    }
                }
                
                await self.send_message(chunk_message)
                
                # 等待分片确认
                chunk_response = await self.receive_message(timeout=60)
                if chunk_response["type"] != "chunk_received":
                    raise Exception(f"分片 {chunk_index} 上传失败")
                
                progress = chunk_response["data"]["progress"]
                print(f"上传进度: {progress:.1f}% ({chunk_index + 1}/{total_chunks})")
        
        print("✓ 所有分片上传完成，等待处理...")
        
        # 等待上传完成通知
        response = await self.receive_message()
        if response["type"] not in ["upload_complete", "task_queued"]:
            raise Exception(f"分片上传完成失败: {response}")
        
        if response["type"] == "task_queued":
            position = response["data"]["queue_position"]
            wait_time = response["data"]["estimated_wait_minutes"]
            print(f"⏳ 任务排队中，位置: {position}，预计等待: {wait_time}分钟")
        
        # 等待转录完成
        return await self._wait_for_result()
    
    async def _wait_for_result(self):
        """等待转录结果"""
        while True:
            response = await self.receive_message(timeout=300)  # 5分钟超时
            
            if response["type"] == "task_progress":
                progress = response["data"]["progress"]
                status = response["data"]["status"]
                message = response["data"].get("message", "")
                print(f"转录进度: {progress}% - {status} - {message}")
            
            elif response["type"] == "task_complete":
                result = response["data"]["result"]
                print("✓ 转录完成")
                return result
            
            elif response["type"] == "error":
                error_msg = response["data"]["message"]
                raise Exception(f"转录失败: {error_msg}")
```

## 性能优化建议

### 客户端优化

1. **文件预处理**
   - 压缩音频文件减少传输时间
   - 使用合适的采样率（16kHz推荐）

2. **网络优化**
   - 使用稳定的网络连接
   - 适当增加超时时间处理大文件

3. **并发控制**
   - 避免同时上传多个大文件
   - 合理设置重试间隔

### 错误恢复

1. **连接断开重试**
   ```python
   async def connect_with_retry(self, max_retry=3):
       for attempt in range(max_retry):
           try:
               return await self.connect()
           except Exception as e:
               if attempt < max_retry - 1:
                   await asyncio.sleep(2 ** attempt)
               else:
                   raise e
   ```

2. **分片重传**
   - 服务器会自动处理重复分片
   - 客户端可以安全地重传失败的分片

## 故障排除

### 常见问题

1. **连接超时**
   - 检查服务器是否正常运行
   - 确认网络连接稳定
   - 适当增加连接超时时间

2. **文件上传失败**
   - 检查文件格式是否支持
   - 验证文件大小是否超过限制
   - 确认文件哈希计算正确

3. **转录结果异常**
   - 检查音频质量
   - 确认音频语言设置
   - 验证文件编码格式

4. **转录中途连接断开**
   - 长音频任务的预期行为，服务端任务不受影响（见「长音频任务与连接断开」一节）
   - 不要重新上传文件，按轮询姿势重连探测结果即可
