# 部署指南

> 本项目仅在 **macOS Apple Silicon** 上运行（依赖 MPS GPU 加速）。

---

## 一、环境要求

| 项 | 要求 |
|---|---|
| 操作系统 | macOS 13+ (Apple Silicon M1/M2/M3) |
| Python | 3.12.x(CapsWriter/qwen_asr_gguf 引擎要求, FunASR parity 已验证 3.12 兼容) |
| 内存 | 8GB 起步, 16GB 推荐(Qwen3 GGUF + Metal ~3.8GB RSS) |
| 磁盘 | 至少 8GB(FunASR 模型 ~1.2GB + Qwen3 模型 ~2.1GB + 缓存数据库 + 日志) |
| 依赖 | FFmpeg（`brew install ffmpeg`） |
| 进程管理 | PM2（**仅 prod 需要**：`npm install -g pm2`） |
| FunASR 模型 | ModelScope 自动下载(~1.2GB), 存 `~/.cache/modelscope/hub/`, prod/dev 共用 |
| Qwen3 模型 | `scripts/download_qwen3_models.sh` 落地到 `models/qwen3_diarize/`(~2.1GB, 仅 default_engine=qwen3 需要; 含 Qwen3-ASR-1.7B GGUF + sherpa pyannote-segmentation + sherpa nemo-titanet-small embedding ~30MB, PR4 cluster_merge 复用同一 embedding 模型, 不额外占空间) |

```bash
pm2 save                    # 保存进程列表，开机自动恢复
pm2 startup                 # 输出一条 sudo 命令，执行后开机自启
```

## 五、引擎切换 (FunASR ↔ Qwen3-Diarize)

服务器是 **全局唯一引擎模式**: 启动时按 `FUNASR_DEFAULT_ENGINE` (或 `config.transcription.default_engine`) 锁定一个引擎, 不支持运行时切换. upload_request.engine 字段必须与之相同或留空, 不同则 reject.

### 切到 Qwen3-Diarize
```bash
# 1) 落地模型 (~2.1GB, 优先从 ~/Production/qwen_asr_server/ + spike 镜像复制)
bash scripts/download_qwen3_models.sh
bash scripts/download_qwen3_models.sh --verify  # 文件就绪校验

# 2) 切配置
echo "FUNASR_DEFAULT_ENGINE=qwen3" >> .env
echo "FUNASR_QWEN3_POOL_SIZE=2" >> .env  # 可选: profile 默认 1 (单实例), 并发需求按内存/显存显式开
# 或编辑 config.json: "transcription.default_engine": "qwen3"

# 3) 重启服务
pm2 restart funasr-server  # 或 venv/bin/python run_server.py
```

启动日志会显示 `Qwen3-Worker-0/1/2 启动` + Metal device init, 各 worker `Qwen3-ASR 引擎加载完成` 即就绪.

### Qwen3 多 Worker 池架构 (PR3, 2026-05-15)

Qwen3 引擎走 **multi-process worker pool**, 跟 FunASR 同一套架构, 但物理隔离:
- 池大小独立: FunASR 用 `max_concurrent_tasks=2`, Qwen3 用 `qwen3_pool_size`（profile 默认 1，互不影响）
- 任务目录物理隔离: FunASR 用 `temp/tasks/`, Qwen3 用 `temp/tasks_qwen3/`(避免同机器多 server 抢任务文件)
- 主进程不持有 libllama context, 由每个 worker subprocess 独立加载 GGUF + Metal + sherpa
- worker 内 libllama context per-process 隔离, 解决并发 unsafe(kv cache / Metal compute buffer 串台)

| 引擎 | pool 字段 | 默认值 | 任务目录 | 内存预算 |
|---|---|---|---|---|
| funasr | `max_concurrent_tasks` | 2 | `temp/tasks/` | ~1-2GB per worker |
| qwen3 | `qwen3_pool_size` | **1**（全 profile，2026-06-10 起） | `temp/tasks_qwen3/` | ~3.5GB per worker (GGUF mmap 共享, N=3 总 ~5-7GB) |

**N 选择建议**(PoC v5 实测, 见 `spikes/qwen3_diarize/spike_report.md` 第 5 节):
- **N=1 默认**（2026-06-10 拍板）: 单任务拿满 GPU 延迟最优; 3060 12GB 实测 N=2 + word_align 双 MMS CUDA session 撞显存 OOM（fallback 不挂但词级时间戳静默丢失）
- N=2 快速验证 / 模型加载减半: 适合 dev 测试 / 内存受限场景（CUDA 上开 word_align 时显存需预算 ≥2×MMS）
- N=3 短任务 throughput 优先: GPU 97% 满载, throughput 2.12 task/min (5min 双人音频, Mac 实测)
- **长音频并发 (>10min) RTF 翻倍预警** (2026-05-16 实测): N=2 双 task 长音频并发时单 task RTF 从 ~0.10 升到 ~0.30-0.43, 因 GPU 在两 task 间分时
- N≥4 GPU 100% 饱和: 纯排队, 不会更快

### Qwen3 num_threads 调优 (PoC 验证)

`FUNASR_QWEN3_NUM_THREADS` 同时控 ONNX (ASR encoder) + sherpa-onnx (segmentation/embedding) 的线程池. PoC 实测 (M1 Max 10 cores, pool_size=2) 默认值 8 → 4 后 N=2 wall -11.5%, 单 task RTF 也更好. 原因: t=8 时总 threads = 2×8 = 16 远超物理核数, CPU 过订阅; t=4 时总 threads = 2×4 = 8 ≤ 10 cores 留 2 core 给系统调度.

**生产默认值**: `FUNASR_QWEN3_NUM_THREADS=4` (已在 .env.example / config.py 同步)

**自定义公式**: `num_threads ≤ (物理核数 - 2) / pool_size`, 例如:
- 8-core CPU + pool_size=2 → num_threads=3
- 10-core CPU + pool_size=2 → num_threads=4 ✅ (生产推荐)
- 16-core CPU + pool_size=3 → num_threads=4 (或 5 但留更少余量)

详见 `spikes/qwen3_mac_hw_accel/SUMMARY.md` / `spikes/qwen3_mac_hw_accel/num_threads_tuning.md`.

### Qwen3 ASR encoder ANE 加速 (macOS, Phase 2)

macOS Apple Silicon 上 **Qwen3-ASR encoder frontend 自动走 CoreML ANE** (从 Phase 2 起默认开启):

- 路由: `build_engine_config` 在 macOS 上默认 `onnx_provider="COREML_ANE_FE"`, 让 frontend ONNX 走 `CoreMLExecutionProvider` + `MLProgram` format + `CPUAndNeuralEngine` units; backend 保 CPU (axis 4 op CoreML 不兼容)
- 收益: PoC 实测 (M1 Max, 60s 双人) frontend 24.5→10.9s, N=2 wall -7.5%, 跟 num_threads=4 组合 -16.1% (495.6→415.8s)
- ANE 启用: mean 50mW max 336mW; llm_decode 保持 37s 不退化 (units 故意排除 GPU 避免抢 llama.cpp Metal)

**关闭/切换** (生产 escape hatch):
```bash
# 强制 CPU (一键回退, 例如 ANE 出问题)
echo "FUNASR_QWEN3_ASR_ENCODER_PROVIDER=cpu" >> .env

# 显式 frontend ANE (默认值, 非 macOS 自动 fallback CPU)
echo "FUNASR_QWEN3_ASR_ENCODER_PROVIDER=coreml_ane_fe" >> .env

# 自动 (按平台感知, 默认值, 也是 unset 时的行为)
echo "FUNASR_QWEN3_ASR_ENCODER_PROVIDER=auto" >> .env
```

**不要做的事** (PoC 验证负面, 详见 `spikes/qwen3_mac_hw_accel/coreml_asr_encoder.md`):
- `MLComputeUnits=ALL` (会抢 llama.cpp Metal, llm_decode +73s)
- `ModelFormat=NeuralNetwork` (silent fallback CPU, 无收益)
- backend 也走 CoreML EP (卡 axis 4 op) — Phase 3 改用 mlpackage 重导, 见下

### Qwen3 ASR encoder backend ANE/GPU (Phase 3, ⚡ N=2 wall -22.6%)

PyTorch → coremltools → `.mlpackage`, backend 走 CoreML (`COREML_ANE_FULL`),frontend 仍 ANE。

**N=2 实测对比** (M1 Max 64GB, macOS 26.5, 16min+44min 长音频并行):

| 指标 | Phase 1 (全 CPU) | Phase 2 (frontend ANE) | **Phase 3 (full ANE/GPU)** |
|---|---|---|---|
| **wall_total** | 495.6s | 458.0s (-7.5%) | **354.2s (-28.5%)** |
| 4spk-44min wall | — | 457.0s | **346.8s (-24.1%)** |
| 4spk encoder total | — | 84.9s | **18.95s (-77.7%)** |
| 4spk backend | 60.7s (CPU) | 60.7s (CPU) | **5.84s (GPU, -90.4%)** |
| 4spk llm_decode | — | 348.2s | **309.5s (-11%, 意外) |

**硬件利用率** (powermetrics 实测 N=2):
- **CPU**: mean 12.6W, peak 32.9W — 主要 sherpa diarize + Python 协调
- **GPU**: mean 16.9W, peak 44.0W, residency 45% — backend mlpackage + LLM decode 共享 Metal
- **ANE**: peak 299mW — frontend ONNX (mean readout 56mW 偏低, macOS 26 powermetrics bug)
- **不再 CPU 打满 GPU 空等**: 详见 `spikes/qwen3_mac_hw_accel/phase3_backend_mlpackage.md`

**启用** (M1 Max 生产可用,N=2 必须配 GPU backend):
```bash
# 1. 生成 .mlpackage (~3.4GB 下载 + 44s 转换, 一次性)
venv/bin/python -m pip install transformers coremltools pyobjc-core pyobjc-framework-CoreML
venv/bin/python spikes/qwen3_mac_hw_accel/phase3_backend/export_backend_coreml.py --time 520

# 2. 切 Phase 3 + N=2 关键 (backend 走 GPU 不抢 ANE)
echo "FUNASR_QWEN3_ASR_ENCODER_PROVIDER=coreml_ane_full" >> .env
echo "FUNASR_QWEN3_BACKEND_MLPACKAGE_UNITS=CPU_AND_GPU" >> .env  # N=2 必须, N=1 可 CPU_AND_NE
```

**为什么 N=2 必须 backend GPU 不 ANE?**

- N=1: frontend ANE + backend ANE 单 worker = 2 路 ANE 推理, **OK**
- N=2: 双 worker = frontend ANE × 2 + backend ANE × 2 = **4 路 ANE 共享**, 触发 llama.cpp `ggml_abort` (LLM sampler 越界)
- 切 backend GPU 后: ANE 只跑 frontend (2 路), GPU 跑 backend + LLM decode (Metal dispatch queue 共享但未观察到冲突)

**实施细节** (避坑):
- 用 PyObjC + `MLMultiArray.initWithDataPointer + deallocator=None` 零拷贝调 CoreML — 绕开 `coremltools.MLModel.predict` 的 `MLE5ExecutionStream` lingering reset SIGSEGV race (macOS 26+ 已知 bug, 详见 `docs/ref/coreml_lingering_race/`)
- 默认 `auto` 仍走 Phase 2 `coreml_ane_fe` (生产稳健); Phase 3 通过显式 env 启用

详细数据 + 5 份 crash report 分析: `spikes/qwen3_mac_hw_accel/phase3_backend_mlpackage.md`

### Qwen3 词级时间戳 (word_align, MMS-300M CTC-FA)

给 Qwen3 引擎**增量挂词级时间戳**到 `segment.words`(不替换段边界)。基于 MMS-300M
CTC-FA(`ctc-forced-aligner` ONNX),多语种。**字段默认关闭;`cuda_prod`/`cuda_dev` profile
默认开**(CUDA 仅 +1% RTF),Mac profile 保持关(CPU +17% RTF)。JSON 输出才带词
(SRT 不带)。实现见 `CLAUDE.md`「Qwen3 后处理 pipeline」5.5 层。

> CUDA 机器用 `FUNASR_PROFILE=cuda_prod`(或 `cuda_dev`)即自动开启,无需手动设
> `WORD_ALIGN_ENABLED`——但**仍须先预下 MMS 模型 + 修 onnxruntime-gpu**(见下)。
> 临时关:`FUNASR_QWEN3_WORD_ALIGN_ENABLED=false`(env 覆盖 profile)。Mac 想开则手动设 true。

```bash
# 1) 预下 MMS 模型 (~1.2GB, 优先拷 ~/ctc_forced_aligner/model.onnx, 缺则 HF 直下)
bash scripts/download_qwen3_models.sh --word-align
#    落到 models/qwen3_diarize/ctc_forced_aligner/model.onnx

# 2) 开启 + 设兜底语言 (ISO 码: chi/eng/jpn/kor...)
echo "FUNASR_QWEN3_WORD_ALIGN_ENABLED=true" >> .env
echo "FUNASR_QWEN3_WORD_ALIGN_LANGUAGE=chi" >> .env   # per-request language 字段优先, 此为兜底
# CUDA 机器额外指定 provider 走 GPU (Mac 留默认 auto→CPU 即可)
echo "FUNASR_QWEN3_WORD_ALIGN_PROVIDER=cuda" >> .env

# 3) 重启服务
pm2 restart funasr-server   # 或 venv/bin/python run_server.py
```

- **per-request 语言**:客户端上传时带 `language` 字段(ISO 码)按请求覆盖;缺省走
  `FUNASR_QWEN3_WORD_ALIGN_LANGUAGE` 兜底。中英混排用 `chi`。
- **RTF 代价**:Mac CPU 约 **+17%**(增量 RTF +0.166),RTX 3060 CUDA 仅 **~+1%**
  (增量 RTF +0.011,2026-06-10 实测,见 `spikes/qwen3_word_timestamp/SUMMARY.md`)。
  CUDA 上可放心常开,Mac CPU 需权衡。
- **失败兜底**:某 chunk 对齐失败 → 该段 `words=null`,段照常出,不崩;遥测在
  `raw_result.word_align`(`failed_windows` / `total_words`)。
- **缓存隔离**:开启后缓存 key 自动变 `qwen3+wa:<lang>`,跟未开启 / 不同语言结果不互相命中。

> ⚠️ **CUDA 机器部署陷阱**:`ctc-forced-aligner` 依赖声明的是 **CPU 版 `onnxruntime`**,
> `pip install -r requirements.txt` 会装上它并**覆盖** `onnxruntime-gpu` 的
> `onnxruntime/capi/*.so` → `CUDAExecutionProvider` 消失,服务启动时
> `_validate_engine_runtime` 直接 fail-fast(`default_engine=qwen3 + runtime=cuda` 报错)。
> **CUDA 机器装完依赖后必须修复**:
> ```bash
> venv/bin/pip uninstall -y onnxruntime
> venv/bin/pip install --force-reinstall --no-deps onnxruntime-gpu==<原版本>   # 如 1.26.0
> # 验证: 带 LD_LIBRARY_PATH 跑 python -c "import onnxruntime;print(onnxruntime.get_available_providers())" 应含 CUDAExecutionProvider
> ```
> Mac/CPU 部署无此问题(本就用 CPU onnxruntime)。MMS CUDA + llama.cpp CUDA + ORT diarize
> 三者进程内共存已验证不 segfault(3060 实测)。

## 六、常见问题

### 模型下载失败
```bash
# 检查网络（ModelScope 在国内访问通常正常）
curl -I https://www.modelscope.cn/

# 手动触发下载
venv/bin/python -c "from modelscope import snapshot_download; \
  snapshot_download('iic/speech_seaco_paraformer_large_asr_nat-zh-cn-16k-common-vocab8404-pytorch')"

# 模型默认存 ~/.cache/modelscope/hub/，可通过 MODELSCOPE_CACHE 环境变量覆盖
```

### 端口被占用
```bash
lsof -i :8767                              # 看占用进程
pm2 list | grep funasr                     # 看是否是另一个 PM2 实例
```

### 数据库迁移失败（PR1 引入）
- PR1 后首次启动会自动 ALTER TABLE 加 `engine` 列
- 失败时检查 `~/Production/funasr_spk_server/data/transcription_cache.db` 是否锁定
- 极端情况备份后删除数据库重建：`mv data/transcription_cache.db data/transcription_cache.db.bak`

### MPS 长音频转录失败
- 已有补丁：`src/core/patches/mps_patch.py`（2025-10-12 引入）
- 详见 [docs/开发/gpu加速/2025-10-12-MPS长音频故障与补丁方案.md](开发/gpu加速/2025-10-12-MPS长音频故障与补丁方案.md)
