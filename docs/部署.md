# 部署指南

> 本项目仅在 **macOS Apple Silicon** 上运行（依赖 MPS GPU 加速）。

---

## 一、环境要求

| 项 | 要求 |
|---|---|
| 操作系统 | macOS 13+ (Apple Silicon M1/M2/M3) |
| Python | 3.12.x(CapsWriter/qwen_asr_gguf 引擎要求, FunASR parity 已验证 3.12 兼容) |
| 内存 | 8GB 起步, 16GB 推荐(Qwen3 GGUF + Metal ~3.8GB RSS) |
| 磁盘 | 至少 8GB(FunASR 模型 ~1.2GB + Qwen3 模型 ~2.1GB + 缓存数据库 + 日志) |
| 依赖 | FFmpeg（`brew install ffmpeg`） |
| 进程管理 | PM2（**仅 prod 需要**：`npm install -g pm2`） |
| FunASR 模型 | ModelScope 自动下载(~1.2GB), 存 `~/.cache/modelscope/hub/`, prod/dev 共用 |
| Qwen3 模型 | `scripts/download_qwen3_models.sh` 落地到 `models/qwen3_diarize/`(~2.1GB, 仅 default_engine=qwen3 需要; 含 Qwen3-ASR-1.7B GGUF + sherpa pyannote-segmentation + sherpa nemo-titanet-small embedding ~30MB, PR4 cluster_merge 复用同一 embedding 模型, 不额外占空间) |

## 二、prod/dev 物理隔离布局

| 维度 | prod | dev |
|---|---|---|
| 目录 | `~/Production/funasr_spk_server/` | `~/Dev/projects/250729_funasr_spk_server/funasr_spk_server/` |
| 端口 | 8767 | 8867 |
| Git 分支 | `main` | `dev`（合并后推 main） |
| Python venv | 项目内 `venv/`（独立） | 项目内 `venv/`（独立） |
| 进程守护 | **PM2**（`funasr-server`） | **无（前台直跑）** |
| 日志级别 | INFO | DEBUG |
| 通知 | 启用企微 webhook | 关闭 |
| 认证 | 启用 | 关闭 |

模型文件不需要复制 — `~/.cache/modelscope/` 是用户级全局缓存，prod/dev 自动共享。

## 三、生产部署（prod）

### 首次部署
```bash
cd ~/Production/
git clone git@github.com:zj1123581321/funasr_spk_server.git
cd funasr_spk_server
git checkout main

# 创建 venv（推荐 uv，也可用 python -m venv）
python3.12 -m venv venv  # 或 uv venv venv --python 3.12
venv/bin/pip install --upgrade pip
venv/bin/pip install -r requirements.txt

# 配置 .env（关键项：webhook、认证密钥、端口）
cp .env.example .env
vim .env  # FUNASR_SERVER_PORT=8767, FUNASR_AUTH_SECRET_KEY=..., FUNASR_WEBHOOK_URL=...

# (可选) 落地 Qwen3 模型权重 — 仅 FUNASR_DEFAULT_ENGINE=qwen3 时需要
bash scripts/download_qwen3_models.sh   # 优先从 ~/Production/qwen_asr_server/models 复制

# 用 PM2 启动(ecosystem.config.cjs 已在仓库里)
pm2 start ecosystem.config.cjs
pm2 save                    # 保存进程列表，开机自动恢复
pm2 startup                 # 输出一条 sudo 命令，执行后开机自启
```

### 日常更新
```bash
cd ~/Production/funasr_spk_server
git pull origin main
venv/bin/pip install -r requirements.txt   # 仅 requirements.txt 有变化时
pm2 restart funasr-server                  # 自动迁移数据库 schema (PR1 后自动)
pm2 logs funasr-server --lines 50          # 验证启动正常
```

### 状态查询 / 重启 / 停止
```bash
pm2 status                  # 看进程状态
pm2 logs funasr-server      # 实时日志
pm2 restart funasr-server   # 重启
pm2 stop funasr-server      # 停止（不删除）
pm2 delete funasr-server    # 删除进程
```

## 五、引擎切换 (FunASR ↔ Qwen3-Diarize)

服务器是 **全局唯一引擎模式**: 启动时按 `FUNASR_DEFAULT_ENGINE` (或 `config.transcription.default_engine`) 锁定一个引擎, 不支持运行时切换. upload_request.engine 字段必须与之相同或留空, 不同则 reject.

### 切到 Qwen3-Diarize
```bash
# 1) 落地模型 (~2.1GB, 优先从 ~/Production/qwen_asr_server/ + spike 镜像复制)
bash scripts/download_qwen3_models.sh
bash scripts/download_qwen3_models.sh --verify  # 文件就绪校验

# 2) 切配置
echo "FUNASR_DEFAULT_ENGINE=qwen3" >> .env
echo "FUNASR_QWEN3_POOL_SIZE=3" >> .env  # 可选: PoC v5 sweet spot 3, 默认 3
# 或编辑 config.json: "transcription.default_engine": "qwen3"

# 3) 重启服务
pm2 restart funasr-server  # 或 venv/bin/python run_server.py
```

启动日志会显示 `Qwen3-Worker-0/1/2 启动` + Metal device init, 各 worker `Qwen3-ASR 引擎加载完成` 即就绪.

### Qwen3 多 Worker 池架构 (PR3, 2026-05-15)

Qwen3 引擎走 **multi-process worker pool**, 跟 FunASR 同一套架构, 但物理隔离:
- 池大小独立: FunASR 用 `max_concurrent_tasks=2`, Qwen3 用 `qwen3_pool_size=3` (互不影响)
- 任务目录物理隔离: FunASR 用 `temp/tasks/`, Qwen3 用 `temp/tasks_qwen3/`(避免同机器多 server 抢任务文件)
- 主进程不持有 libllama context, 由每个 worker subprocess 独立加载 GGUF + Metal + sherpa
- worker 内 libllama context per-process 隔离, 解决并发 unsafe(kv cache / Metal compute buffer 串台)

| 引擎 | pool 字段 | 默认值 | 任务目录 | 内存预算 |
|---|---|---|---|---|
| funasr | `max_concurrent_tasks` | 2 | `temp/tasks/` | ~1-2GB per worker |
| qwen3 | `qwen3_pool_size` | 3 | `temp/tasks_qwen3/` | ~3.5GB per worker (GGUF mmap 共享, N=3 总 ~5-7GB) |

**N 选择建议**(PoC v5 实测, 见 `spikes/qwen3_diarize/spike_report.md` 第 5 节):
- N=2 快速验证 / 模型加载减半: 适合 dev 测试 / 内存受限场景
- N=3 生产推荐: GPU 97% 满载, throughput 2.12 task/min (5min 双人音频)
- **长音频并发 (>10min) RTF 翻倍预警** (2026-05-16 实测): N=2 双 task 长音频并发时单 task RTF 从 ~0.10 升到 ~0.30-0.43, 因 GPU 在两 task 间分时. 长任务延迟敏感场景建议 N=1 拿满 GPU, 短任务 throughput 优先场景仍 N=3
- N≥4 GPU 100% 饱和: 纯排队, 不会更快

### Qwen3 num_threads 调优 (PoC 验证)

`FUNASR_QWEN3_NUM_THREADS` 同时控 ONNX (ASR encoder) + sherpa-onnx (segmentation/embedding) 的线程池. PoC 实测 (M1 Max 10 cores, pool_size=2) 默认值 8 → 4 后 N=2 wall -11.5%, 单 task RTF 也更好. 原因: t=8 时总 threads = 2×8 = 16 远超物理核数, CPU 过订阅; t=4 时总 threads = 2×4 = 8 ≤ 10 cores 留 2 core 给系统调度.

**生产默认值**: `FUNASR_QWEN3_NUM_THREADS=4` (已在 .env.example / config.py 同步)

**自定义公式**: `num_threads ≤ (物理核数 - 2) / pool_size`, 例如:
- 8-core CPU + pool_size=2 → num_threads=3
- 10-core CPU + pool_size=2 → num_threads=4 ✅ (生产推荐)
- 16-core CPU + pool_size=3 → num_threads=4 (或 5 但留更少余量)

详见 `spikes/qwen3_mac_hw_accel/SUMMARY.md` / `spikes/qwen3_mac_hw_accel/num_threads_tuning.md`.

### Qwen3 引擎 preset 调优
默认 PoC `preset=auto` (普适最优, 自适应聚类). 已知说话人数时锁簇可更精准:
```bash
# 已知 2 人 (固定簇)
echo "FUNASR_QWEN3_NUM_SPEAKERS=2" >> .env

# 未知人数 (默认自适应, threshold 0.9)
echo "FUNASR_QWEN3_NUM_SPEAKERS=auto" >> .env

# 阈值微调 (越大聚出的 speaker 越少, 默认 0.9)
echo "FUNASR_QWEN3_CLUSTER_THRESHOLD=0.85" >> .env
```

详细 preset 对照见 `spikes/qwen3_diarize/spike_report.md` 第 4 节.

## 六、常见问题

### 模型下载失败
```bash
# 检查网络（ModelScope 在国内访问通常正常）
curl -I https://www.modelscope.cn/

# 手动触发下载
venv/bin/python -c "from modelscope import snapshot_download; \
  snapshot_download('iic/speech_seaco_paraformer_large_asr_nat-zh-cn-16k-common-vocab8404-pytorch')"

# 模型默认存 ~/.cache/modelscope/hub/，可通过 MODELSCOPE_CACHE 环境变量覆盖
```

### 端口被占用
```bash
lsof -i :8767                              # 看占用进程
pm2 list | grep funasr                     # 看是否是另一个 PM2 实例
```

### 数据库迁移失败（PR1 引入）
- PR1 后首次启动会自动 ALTER TABLE 加 `engine` 列
- 失败时检查 `~/Production/funasr_spk_server/data/transcription_cache.db` 是否锁定
- 极端情况备份后删除数据库重建：`mv data/transcription_cache.db data/transcription_cache.db.bak`

### MPS 长音频转录失败
- 已有补丁：`src/core/patches/mps_patch.py`（2025-10-12 引入）
- 详见 [docs/开发/gpu加速/2025-10-12-MPS长音频故障与补丁方案.md](开发/gpu加速/2025-10-12-MPS长音频故障与补丁方案.md)
